34  慢节点、故障与恢复

34.1 本章要回答的问题

分布式 AI 最大的现实问题之一是：整体速度由最慢 rank 决定，整体可靠性受任意关键 rank 影响。

单卡任务失败，通常是一个进程失败。分布式任务失败，可能是一个 rank OOM、一个网络连接超时、一个节点 GPU 降频、一个 shard checkpoint 写坏，最后表现为整个任务卡住或退出。

34.2 什么是 straggler

straggler 指拖慢整体进度的慢 rank、慢 stage、慢副本或慢节点。

在 DDP 中，某个 rank backward 慢，其他 rank 会在 AllReduce 附近等待。在 PP 中，某个 stage 慢，后续 stage 拿不到 activation。在分布式推理中，某个 TP group 内一张 GPU 慢，整个模型副本的 decode 速度都会下降。

分布式系统里，平均值经常掩盖问题。你看到平均 step time 是 1 秒，但可能 15 个 rank 是 800 ms，1 个 rank 是 1 秒，所有人都被它拖到 1 秒。

34.3 慢节点从哪里来

常见来源可以按资源流拆开。

来源 表现 例子
数据慢 rank 等 batch 某个节点读取远端存储慢，tokenizer CPU 不够
计算慢 某 rank forward/backward 更慢 GPU 降频、MIG 配置不同、输入长度不同
显存问题 OOM、频繁释放、cache eviction batch 太大，KV Cache 逼近上限
网络慢 collective 或 stage send/recv 变慢 RDMA 配置异常、跨机架、拥塞
存储慢 checkpoint 卡住 多 rank 同时写对象存储
代码路径不一致 某些 rank 执行额外逻辑 rank 0 logging 过重，验证逻辑放错位置

这也是为什么本书前面先讲五条资源流：CPU、GPU、显存、网络、存储。straggler 往往不是“GPU 慢”这一个词能解释的。

34.4 故障类型

分布式 AI 常见故障包括：

故障 表现 影响
rank 进程退出 某个 worker crash 训练通常整体失败或 hang
GPU OOM 某个 rank CUDA OOM 可能导致 communicator 异常
NCCL timeout collective 长时间无响应 多数 rank 等待，最终超时
节点重启 多个 rank 消失 需要整组恢复
checkpoint 损坏 恢复失败 训练进度丢失或需要回滚
推理副本半失败 服务还活着但生成异常 P99 变差，流式请求中断

只看进程存活不够。很多问题不是“进程死了”，而是进程活着但不再有效推进。

34.5 观测要到 rank 级别

分布式排障需要把指标打到 rank、stage 或 replica group。

指标 用途
per-rank step time 找最慢 rank
DataLoader time 判断是否数据供给慢
forward/backward time 判断计算是否不均
collective time 判断通信是否成为瓶颈
GPU utilization / HBM usage 判断计算和显存状态
network throughput / retransmit 判断网络异常
checkpoint save/load time 判断存储是否拖慢
inference TTFT/TPOT by replica 判断推理慢副本

这些指标要能按 rank 关联到节点、GPU、网卡、Pod 和任务 ID。否则看到 NCCL timeout，也不知道是哪条链路、哪个节点、哪个 rank 引起的。

34.6 恢复不是简单重启

普通服务重启后还能继续处理新请求。训练任务重启后，如果没有 checkpoint，可能丢失数小时甚至数天计算。

训练恢复至少涉及：

  • model parameters。
  • optimizer state。
  • scheduler state。
  • global step。
  • random state。
  • dataset sampler state。
  • sharded checkpoint 元数据。
  • 训练配置和代码版本。

如果只保存 weights，通常只能用于推理或 fine-tune 起点，不等于可无缝恢复训练。长训练任务必须把 checkpoint 当成系统设计的一部分，而不是训练脚本最后的附属功能。

34.7 gang restart 为什么常见

很多分布式训练任务要求所有 rank 处于一致状态。某个 rank 失败后，其他 rank 未必能继续独自推进。最常见的恢复方式是整组停止，从最近 checkpoint 重新启动。

这叫 gang restart 的直觉：

一个 rank 失败
-> 整个 worker group 停止
-> 从一致 checkpoint 恢复
-> 所有 rank 重新 rendezvous
-> 继续训练

它看起来粗暴，但比让剩余 rank 带着不一致状态继续跑更可靠。Elastic training 可以在某些场景动态改变 worker 数，但它有前提，并不是所有模型、数据管线和并行策略都天然支持。

34.8 推理故障恢复的不同点

推理没有 global step，但有在线请求状态。尤其是流式 LLM 请求，故障可能发生在已经返回了一部分 token 之后。

推理恢复要回答：

  • 客户端连接断了，runtime 是否释放 KV Cache？
  • 副本 OOM 后，队列里的请求是否重试？
  • 重试是否会导致重复生成或顺序变化？
  • prefix cache 是否跨副本共享？
  • prefill/decode 分离时，decode 失败是否要重跑 prefill？
  • 网关是否能把慢副本摘掉，而不是继续分流？

训练故障关注“进度能不能恢复”；推理故障关注“请求能否按 SLA 完成或明确失败”。

34.9 一个 NCCL timeout 案例

现象：多节点 DDP 训练运行数小时后卡住，日志出现 NCCL timeout。

可能原因不是一个：

方向 检查
慢 rank 查看 timeout 前各 rank step time 是否分化
网络 查看 RDMA、网卡错误、跨节点链路、拥塞
数据 某个 rank 是否在 DataLoader 阻塞，导致没进入 collective
OOM 某个 rank 是否先 CUDA OOM，其他 rank 等它
checkpoint timeout 是否总发生在保存点附近
拓扑 任务是否被调度到跨机架或弱链路节点

NCCL timeout 是症状，不一定是根因。真正排障要把 rank timeline、节点拓扑和资源指标拼起来。

34.10 平台治理能力

为了让分布式 AI 在集群里稳定运行，平台需要：

  • gang scheduling：成组调度，避免 rank 缺失。
  • topology-aware placement：根据并行策略选择 GPU 和节点。
  • health check：区分进程存活和训练有效推进。
  • checkpoint policy：控制保存频率、保留策略、恢复路径。
  • failure domain：避免把关键 rank 都放在同一高风险节点或机架。
  • quarantine：识别频繁出错或性能异常的节点。
  • preemption policy：抢占前触发 checkpoint 或优雅终止。
  • rank-level observability：让排障能定位到 rank、GPU、NIC 和存储路径。

这些能力会在后面的 Kubernetes、调度与资源优化部分变成具体实现。

34.11 本章小结

分布式 AI 的故障治理要记住两句话：性能由最慢 rank 决定，恢复由一致状态决定。 straggler 排障要拆资源流，故障恢复要依赖 checkpoint、gang restart、拓扑和 rank 级观测。理解这些，再看 K8s 上的 AI 任务部署，才知道平台真正要解决什么。
