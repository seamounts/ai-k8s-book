29  Data Parallel 与 DDP

29.1 本章要回答的问题

Data Parallel 是最常见的分布式训练策略。它的直觉很朴素：每张 GPU 放一份完整模型，喂不同数据，最后把 gradients 对齐。

PyTorch DistributedDataParallel, 简称 DDP, 是 PyTorch 官方推荐的多进程数据并行方式之一 (《DistributedDataParallel》, 不详)。它不会自动让模型变小，也不会自动解决数据加载慢的问题。它解决的是：当模型单卡能放下时，如何用多张 GPU 加快训练吞吐。

29.2 Data Parallel 的基本形态

单卡训练时，一个 mini-batch 进入一份模型：

batch -> model -> loss -> gradients -> optimizer step

Data Parallel 中，global batch 会被切给多个 rank：

rank 0: batch shard A -> model replica -> gradients A
rank 1: batch shard B -> model replica -> gradients B
rank 2: batch shard C -> model replica -> gradients C
rank 3: batch shard D -> model replica -> gradients D

AllReduce gradients -> 每个 rank 得到一致 gradients -> 各自更新 parameters

这里有三个关键词：

  • replica：每张 GPU 上都有完整 parameters。
  • data shard：每个 rank 只看当前 step 的一部分数据。
  • AllReduce：把多个 rank 的 gradients 聚合并分发回每个 rank。

DDP 的正确性来自一个假设：如果每个 rank 用不同数据算出 gradients，再对 gradients 求和或平均，那么它近似等价于用更大的 batch 在一张逻辑 GPU 上训练。

29.3 global batch 怎么算

多卡训练最容易被忽略的是 batch 语义。常用公式是：

global batch size =
  per-device batch size
  x gradient accumulation steps
  x world size

假设每张 GPU 的 per-device batch size 是 8，world size 是 16，gradient accumulation steps 是 4，那么 global batch size 是：

8 x 16 x 4 = 512

这不是一个纯系统参数，它会影响训练收敛。把 GPU 从 8 张扩到 64 张，如果 per-device batch size 不变，global batch size 会放大 8 倍。此时可能需要重新考虑 learning rate、warmup、optimizer 和数据 shuffle。

29.4 DistributedSampler 为什么重要

分布式训练里，不能让每个 rank 都按同样顺序读完整数据。否则看起来用了多张 GPU，实际每张卡在重复看同一批样本，global batch 的语义会错。

PyTorch DistributedSampler 用于把 dataset 切给多个进程，让每个 rank 读取自己的数据 shard (《torch.utils.data.distributed.DistributedSampler》, 不详)。它解决的是数据分发的正确性，而不仅是性能。

一个常见错误是：训练脚本能跑，loss 也在下降，但多个 rank 重复读取相同数据。最后模型看似训练了很多 step，实际有效样本数远低于预期。这类问题从 GPU 利用率上不一定能看出来，必须检查 sampler、epoch seed 和数据划分逻辑。

29.5 AllReduce 是 DDP 的核心同步点

在 backward 过程中，每个 rank 得到本地 gradients。DDP 需要把这些 gradients 同步成一致结果。NCCL 提供 GPU 间高效 collective 通信，AllReduce 是其中最常见的操作之一 (《NVIDIA NCCL User Guide》, 不详)。

AllReduce 可以理解为两件事合在一起：

  1. Reduce：把多个 rank 的 gradients 聚合起来。
  2. Broadcast：把聚合后的结果发回所有 rank。

DDP 的效率取决于计算和通信是否能较好重叠。模型参数很多、网络慢、拓扑差，或者某个 rank backward 特别慢，都会让其他 rank 在同步点等待。

29.6 什么时候 DDP 很合适

DDP 适合以下条件同时成立的场景：

条件 说明
模型能完整放入单卡 每个 rank 都有完整 replica
单 step 计算量较大 有足够计算覆盖通信成本
数据加载能跟上 多卡会放大数据供给压力
网络和拓扑稳定 AllReduce 对网络抖动敏感
global batch 可接受 扩卡后 batch 语义仍符合训练目标

例如 ResNet、BERT 类模型在常规规模下很适合 DDP。模型完整放得下，每个 step 的矩阵计算足够多，gradient 同步成本可控。

29.7 什么时候 DDP 不够

DDP 不适合把所有问题都解决掉。

问题 为什么 DDP 不够 可能方向
模型 parameters 单卡放不下 DDP 每卡都有完整 replica Tensor Parallel、FSDP、ZeRO-3
optimizer state 太大 每卡通常仍有完整 optimizer state ZeRO、FSDP
batch 放大影响收敛 多卡改变 global batch 调 learning rate、gradient accumulation、训练策略
数据读取成为瓶颈 多卡同时消费数据 优化 DataLoader、缓存、数据格式、存储
跨节点网络弱 AllReduce 时间变长 拓扑放置、RDMA、减少通信、换并行策略

DDP 的边界要说清楚：它是吞吐扩展工具，不是显存压缩工具。

29.8 一个排障案例

现象：8 张 GPU 跑 DDP，Pod 都 Running，GPU 利用率只有 20%，step time 抖动明显。

不要先怀疑 DDP 本身。按训练 step 拆开看：

检查点 可能发现
DataLoader 时间 tokenizer 或数据读取太慢，rank 等数据
每个 rank 的 batch 数 sampler 配错，某些 rank 数据更少或重复
backward 时间 某些输入更长，activation 更多，导致慢 rank
AllReduce 时间 网络或拓扑差，通信拖慢
checkpoint 时间 所有 rank 在保存点同时写存储

DDP 的关键特征是同步。只要一个 rank 慢，其他 rank 就会在 collective 或 barrier 附近等它。排障要看 rank 级别的 step timeline，而不是只看平均 GPU 利用率。

29.9 平台需要为 DDP 提供什么

后面进入 Kubernetes 时，DDP 会变成一组互相依赖的进程。平台至少要表达：

  • world size：总共有多少 rank。
  • rank 编号：每个进程知道自己是第几个 rank。
  • rendezvous 地址：rank 之间如何发现彼此。
  • 同启同停：少一个 rank，任务可能无法正常开始或继续。
  • 网络连通性：rank 间端口、DNS、Service、HostNetwork 或容器网络策略。
  • 拓扑偏好：通信密集任务尽量放在高速互联内。
  • 统一配置：所有 rank 使用相同代码、镜像、模型配置和数据版本。

这就是为什么 AI 训练平台常常需要 Training Operator、VolcanoJob、Kueue 或自定义控制器。普通 Job 能启动进程，但不一定理解 rank 之间的集体语义。

29.10 本章小结

DDP 的一句话本质是：复制模型，切分数据，同步 gradients。 它适合模型能放下、想提高训练吞吐的场景。DDP 的系统瓶颈通常出现在数据供给、AllReduce、慢 rank 和 global batch 语义上。理解 DDP 后，再看 FSDP、ZeRO、Tensor Parallel 和 Pipeline Parallel，就会清楚它们到底是在补 DDP 的哪一个短板。
