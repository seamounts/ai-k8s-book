33  分布式推理

33.1 本章要回答的问题

训练分布式围绕 global step；推理分布式围绕在线请求。它们都用 GPU 和通信，但目标不同。

训练关心的是 step time、吞吐、收敛和 checkpoint；推理关心的是 TTFT、TPOT、P95/P99、吞吐、可用性、成本和用户体感。LLM 推理还要管理 KV Cache、continuous batching、长上下文和流式返回。

vLLM 的 serve CLI 和 ParallelConfig 提供了 tensor parallel、pipeline parallel、data parallel 等多 GPU 服务相关配置 (《vLLM Serve CLI and Parallel Configuration》, 不详)。同时，vLLM 也有 disaggregated prefill 示例，把 prefill 和 decode 作为可拆分阶段来组织 (《vLLM Disaggregated Prefill Example》, 不详)。本章从系统本质解释这些策略。

33.2 推理扩展的三条路线

分布式推理先分清三件事。

路线 切的对象 适合解决 主要难点
多副本 请求流量 总 QPS 不够、需要高可用 路由、冷启动、版本一致性、负载均衡
模型并行 单个模型副本 模型太大、单副本太慢 TP/PP 通信、拓扑、成组调度
阶段拆分 prefill/decode 或 embedding/rerank/generate 不同阶段资源形态不同 KV Cache 传递、队列协调、SLA 拆解

多副本像开多个服务窗口；模型并行像多个人一起服务同一个窗口；阶段拆分像把“接单、加工、出货”拆成不同工位。三者可以组合，但不能混为一谈。

33.3 多副本：最直观，但不总够

最简单的扩展是多跑几个模型副本：

Gateway -> Replica A
        -> Replica B
        -> Replica C

它适合模型单副本能放下、单副本延迟可接受、只是总流量不够的场景。平台可以用常规服务治理能力做负载均衡、健康检查、灰度和回滚。

但 LLM 推理里，多副本会遇到新的约束：

  • 每个副本都要加载一份 weights，显存成本高。
  • 长上下文请求会占用大量 KV Cache。
  • continuous batching 让副本内部排队状态变得重要。
  • 冷启动要下载和加载大模型，扩容不是秒级完成。
  • 路由如果不看队列和 KV Cache，只看连接数，可能负载不均。

所以推理副本不是普通 Web 服务副本。它的容量由 weights、KV Cache、batch 策略和请求形态共同决定。

33.4 模型并行：一个副本跨多张 GPU

当单张 GPU 放不下模型，或者单副本 decode 太慢，可以用 Tensor Parallel 或 Pipeline Parallel 把一个模型实例切到多张 GPU。

推理 TP 的特点：

  • 多张 GPU 共同承载一个模型副本。
  • 每次 token decode 都可能发生层内通信。
  • GPU 必须成组调度，少一张就不是完整副本。
  • 拓扑会直接影响 TPOT。

推理 PP 的特点：

  • 不同层放到不同 stage。
  • activation 在 stage 之间流动。
  • 对超大模型或离线批推理有价值。
  • 在线低延迟场景要谨慎处理 pipeline bubble 和串行依赖。

一句话区分：多副本扩的是服务窗口数量；TP/PP 扩的是一个窗口背后的协同能力。

33.5 prefill/decode 分离

LLM 请求有两个阶段：

  • prefill：把 prompt 一次性处理完，建立初始 KV Cache。长 prompt 时计算量大，更像大块矩阵计算。
  • decode：逐 token 生成，每一步读取历史 KV Cache，追加新 token。它更像小步循环，容易受 KV Cache 访问和 batch 形态影响。

前面第三篇已经讲过 prefill 和 decode 的区别。本章把它放到分布式推理里看：两者资源形态不同，就可能拆到不同 GPU 池或不同服务层。

prefill/decode 分离的收益可能包括：

  • 长 prompt 不阻塞短 decode。
  • prefill worker 针对大块计算优化。
  • decode worker 针对低延迟和 KV Cache 访问优化。
  • 不同阶段独立扩缩容。

但它的代价也很明确：

  • KV Cache 或中间状态需要跨阶段传递。
  • 路由必须知道请求处于哪个阶段。
  • 队列策略变成多段队列。
  • 故障时要处理半完成请求。

这不是“把服务拆两个 Deployment”那么简单，而是把一次请求的生命周期拆成多个状态转移。

33.6 KV Cache 让路由变成状态问题

普通 HTTP 服务的路由通常可以无状态：请求来，分给任意健康副本。LLM 推理不总是这样。

如果一个请求已经在某个副本上生成了 KV Cache，后续 decode 最好继续在能访问这份 KV Cache 的位置执行。否则要么重算 prompt，要么迁移 cache。vLLM 的 PagedAttention 和 Prefix Caching 都是在围绕 KV Cache 做更细粒度的显存管理和复用 (《vLLM Paged Attention》, 不详; 《vLLM Automatic Prefix Caching》, 不详)。

因此分布式推理的路由要考虑：

  • 模型版本和 adapter。
  • 当前副本的队列长度。
  • 副本的 KV Cache 水位。
  • 请求上下文长度和 max tokens。
  • 是否命中 prefix cache。
  • 租户优先级和 SLA。

路由如果只看 QPS 或连接数，就看不到真正的瓶颈。

33.7 SLA 约束下的分布式推理

推理扩展必须用指标约束，不然容易只优化吞吐。

指标 说明 分布式影响
TTFT 请求到首 token 的时间 路由、排队、tokenizer、prefill、跨阶段传递
TPOT 后续每 token 平均时间 decode batch、TP 通信、KV Cache 访问
P95/P99 长尾延迟 长请求、慢副本、队列策略、故障重试
tokens/sec 总生成吞吐 batching、GPU 利用率、模型并行
cache hit rate prefix 或 KV 复用命中率 路由和 prompt 结构
eviction / OOM KV Cache 容量压力 batch、上下文长度、并发

推理平台最重要的设计原则是：吞吐优化必须受延迟目标约束。 为了 tokens/sec 把 batch 做大，却让 TTFT 和 P99 崩掉，是错误优化。

33.8 一个容量规划例子

假设一个 70B LLM 服务有三类请求：

  • 交互式短 prompt，要求 TTFT 低。
  • 长文档总结，prompt 很长。
  • 离线批量生成，对延迟不敏感。

一种合理的分布式推理架构可能是：

请求类型 推荐策略
交互式短 prompt 独立低延迟副本池，较小 queue delay，限制 max tokens
长文档总结 prefill 能力更强的副本池，按上下文长度分桶
离线批量生成 大 batch 或离线队列，优先吞吐

如果三类请求混在一个队列里，长 prompt 会拖慢短请求，离线批任务会挤占交互式 SLA。这不是模型问题，而是服务调度问题。

33.9 平台需要提供什么

分布式推理平台要表达的不只是 Deployment 副本数：

  • 模型副本的 GPU group，例如 TP size。
  • 模型版本、tokenizer、template、adapter 的一致性。
  • KV Cache 资源水位。
  • 请求队列、优先级、取消和超时。
  • prefill/decode 阶段拓扑。
  • 冷启动、warmup 和模型加载进度。
  • SLA 指标和自动扩缩容信号。

后面进入 KServe、vLLM、Ray Serve 和 Kubernetes 推理架构时，会把这些能力落到具体组件上。

33.10 本章小结

分布式推理的本质是：在 SLA 约束下，把请求、模型副本、KV Cache 和计算阶段组织起来。 多副本解决总吞吐，TP/PP 解决单副本容量和速度，prefill/decode 分离解决不同阶段资源形态差异。推理系统不能只看 Pod 存活，要看请求生命周期和显存状态。
