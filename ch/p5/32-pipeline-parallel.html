32  Pipeline Parallel

32.1 本章要回答的问题

Pipeline Parallel, 简称 PP, 切的是模型层。它把一个深模型拆成多个 stage，每个 stage 放到不同 GPU 或节点上。输入先经过 stage 1，再传给 stage 2，最后到 stage N。

PyTorch Pipeline Parallel 教程把 pipeline parallelism 用于把模型按层切分到多个设备，并通过 micro-batch 让多个 stage 尽量同时工作 (《Getting Started with Distributed Pipeline Parallelism》, 不详)。直觉上，它像一条生产流水线：前段加工完半成品，交给下一段继续加工。

32.2 PP 切的是什么

一个 Transformer 有很多层。PP 把这些层分段：

Stage 0: Layer 0-7
Stage 1: Layer 8-15
Stage 2: Layer 16-23
Stage 3: Layer 24-31

forward 时，stage 0 计算 activation，发给 stage 1；stage 1 继续计算，再发给 stage 2。backward 时，gradient 反方向传回去。

PP 解决的是两个问题：

  • 单卡放不下完整模型层。
  • 模型太深，希望让不同 layer stage 并行工作。

32.3 micro-batch 和 pipeline bubble

如果只给流水线一个大 batch，stage 0 算完后 stage 1 才开始，其他 stage 大量时间都在等。为了让多个 stage 同时工作，需要把 batch 再切成多个 micro-batch。

简化后可以这样看：

时间片 1: Stage 0 处理 MB1
时间片 2: Stage 0 处理 MB2, Stage 1 处理 MB1
时间片 3: Stage 0 处理 MB3, Stage 1 处理 MB2, Stage 2 处理 MB1

流水线刚开始和结束时，总有 stage 空着，这部分空闲叫 pipeline bubble。micro-batch 越少，bubble 占比越高；micro-batch 越多，调度更满，但 activation、通信和调度开销也可能增加。

32.4 PP 的通信是什么

PP 的通信不是 gradients AllReduce，也不是层内矩阵 AllGather。它主要传递：

  • forward 方向的 activation。
  • backward 方向的 activation gradient。

这类通信发生在相邻 stage 之间。它不像 TP 那样每层内部频繁同步，但 stage 切分不好时，activation 传输也会成为瓶颈。

32.5 stage 均衡比想象中重要

PP 的吞吐由最慢 stage 决定。如果 stage 0 放了 4 层，stage 1 放了 12 层，stage 1 就会成为瓶颈。其他 stage 计算完后只能等它。

stage 不均衡可能来自：

  • 层数不均。
  • 某些层更重，例如 attention 或 MLP 维度更大。
  • 某些 stage activation 更大，通信更慢。
  • 某些 GPU 型号、温度、功耗或网络路径不同。
  • checkpoint 或 logging 集中在某个 stage。

所以 PP 切分不是平均分层那么简单，而是要按计算、显存和通信一起切。

32.6 1F1B 的直觉

训练 pipeline 常见调度会让一个 stage 交替执行 forward 和 backward，例如 1F1B, 即 one-forward-one-backward。它的目标是减少 activation 常驻时间，并让 stage 更持续地工作。

不用一开始陷入调度细节。记住两个原则：

  1. micro-batch 用来填流水线。
  2. 调度策略用来减少等待和显存压力。

如果 pipeline bubble 大，GPU 利用率低，可能不是 GPU 不够，而是 micro-batch、stage 切分和调度策略没有配好。

32.7 PP、TP、FSDP 可以组合

大模型训练常常不是只用一种并行策略：

组合 直觉
PP + TP 每个 stage 内再做 tensor parallel，承载更大的层
PP + DDP 多条 pipeline 处理不同数据 shard，提高吞吐
PP + FSDP stage 内训练状态再分片，降低显存
PP + activation checkpointing 少存 activation，用重算换显存

这就是所谓 3D parallelism 的来源：Data Parallel、Tensor Parallel、Pipeline Parallel 从不同维度切模型和数据。平台工程最终要能表达这些组关系，而不只是“申请 N 张 GPU”。

32.8 推理中是否需要 PP

推理也可以使用 PP，尤其当模型层太多或单卡放不下时。但 LLM 在线推理有逐 token decode，pipeline stage 之间的串行依赖会影响 latency。对于低延迟交互式服务，TP 往往比 PP 更常见；对于超大模型或离线批推理，PP 仍然有价值。

推理 PP 要特别关注：

  • stage 间 activation 传输是否影响 TPOT。
  • 长短请求混在一起时，pipeline 是否被长请求拖住。
  • prefill 和 decode 是否适合放在同一种 pipeline 策略下。
  • stage 失败时，流式请求如何恢复或失败返回。

32.9 平台需要表达 stage

PP 给平台带来的是 stage 语义：

  • 哪些 rank 属于同一条 pipeline。
  • 每个 stage 需要多少 GPU、显存、CPU 和网络。
  • stage 间是否必须靠近放置。
  • stage 不均衡时如何观测和告警。
  • 重启时是否整条 pipeline 一起重启。
  • checkpoint 是否按 stage 和 shard 保存。

如果平台只看到一堆 Pod，就很难知道哪个 Pod 是 stage 0，哪个是 stage 3，更难判断为什么整条 pipeline 吞吐低。

32.10 一个排障案例

现象：4-stage PP 训练中，GPU 0 利用率高，GPU 1 和 GPU 2 中等，GPU 3 经常空等。

可能原因：

线索 解释
stage 0 时间最长 前几层或数据预处理集中在 stage 0
stage 3 等待多 上游 activation 供给不稳定
micro-batch 数少 pipeline bubble 大
stage 之间网络慢 activation 传输拖慢
checkpoint 在最后 stage 保存时最后 stage 阻塞

排障思路不是“把 GPU 加倍”，而是拆 stage timeline，看每个 stage 的 compute、send、recv、wait 和 checkpoint 时间。

32.11 本章小结

Pipeline Parallel 的一句话本质是：把模型层切成 stage，用 micro-batch 填满流水线。 它适合模型太深、单卡放不下或需要多阶段协同计算的场景。PP 的关键问题是 bubble、stage 均衡、activation 通信和故障恢复。
