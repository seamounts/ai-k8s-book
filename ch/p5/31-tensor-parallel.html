31  Tensor Parallel

31.1 本章要回答的问题

Data Parallel 是切数据，FSDP/ZeRO 是切训练状态。Tensor Parallel, 简称 TP, 切的是更细的东西：一层里面的 tensor 计算。

当模型的一层太大，或者单个模型副本推理太慢时，可以把同一层的矩阵乘、attention head 或 MLP 计算切到多张 GPU 上。PyTorch Tensor Parallel 文档把 TP 描述为在模型层内部切分模块或参数的并行方式 (《Tensor Parallelism》, 不详)。Megatron-LM 则是大模型张量并行和模型并行的重要代表工作 (Shoeybi 等 2019)。

31.2 为什么要切一层

Transformer 里每一层都有 attention 和 MLP，大量计算可以归结为矩阵乘。模型越大，单层 weight matrix 越大，hidden size 越大，attention head 越多。

如果一层的 weights 或中间结果单卡放不下，或者单卡处理这一层太慢，就需要把层内计算拆开。TP 的直觉是：

一个大矩阵乘：
  X @ W

切成多个局部矩阵乘：
  GPU 0: X @ W0
  GPU 1: X @ W1
  GPU 2: X @ W2

再通过通信得到完整输出。

这和 DDP 完全不同。DDP 中每个 rank 都有完整层，处理不同数据；TP 中多个 rank 合作计算同一层，通常处理同一批数据的不同 tensor 分片。

31.3 列切、行切和 head 切

TP 常见切法包括：

切法 直觉 通信特点
Column Parallel 按输出维度切 weight matrix 每张卡产生一部分输出，后面可能需要 AllGather
Row Parallel 按输入维度切 weight matrix 每张卡产生局部和，通常需要 AllReduce
Attention Head Parallel 不同 GPU 负责不同 attention head head 之间相对独立，但后续投影可能通信

不用一开始记公式，先抓住本质：TP 把一层拆成多个局部算子，局部结果必须在某些边界上重新合并。合并就是通信。

31.4 TP 为什么强依赖拓扑

TP 的通信发生在层内部。Transformer 有很多层，每层 forward 和 backward 都可能通信。和 DDP 相比，TP 的通信频率更高。

策略 通信发生位置 拓扑敏感度
DDP backward gradients 同步，通常每个 step 多次 bucket 通信 中到高
FSDP/ZeRO module 前后聚合和切分状态 高
Tensor Parallel 层内矩阵计算边界 很高

这就是为什么 TP 通常优先放在同一节点的 NVLink/NVSwitch 内。跨节点做 TP 不是绝对不能做，但如果链路慢，层内通信会反复惩罚每个 token、每个 step。

31.5 TP 在训练和推理中的差异

训练 TP 要处理 forward、backward、gradient 和 optimizer。它经常和 DDP、FSDP、Pipeline Parallel 组合，形成 2D/3D parallelism。

推理 TP 更常见于大模型服务：单张 GPU 放不下完整模型，或者单副本 decode 太慢，就把一个模型实例切到多张 GPU。vLLM 的分布式 serving 支持 tensor parallel size 等配置，用于把模型部署到多 GPU 上 (《vLLM Serve CLI and Parallel Configuration》, 不详)。

推理 TP 的关键不是只有吞吐，还包括：

  • 首 token 延迟是否变差。
  • TP 通信是否拉高每 token 时间。
  • KV Cache 是否也要按 TP 组管理。
  • 单副本多卡占用是否降低了集群可调度性。

31.6 TP 和 FSDP/ZeRO 的区别

二者都可能让大模型跑起来，但切法不同。

维度 FSDP / ZeRO Tensor Parallel
切分对象 训练状态 shard 层内 tensor / parameter 维度
主要目标 降低训练状态显存 让单层计算和模型副本跨多卡
通信位置 module 边界、状态聚合和切分 层内部，频率更高
推理是否常用 FSDP 主要用于训练 TP 是大模型推理常用手段
拓扑要求 高 很高

简单说：FSDP/ZeRO 更像“账本分片”，TP 更像“同一台机器的多个车间同时加工一个零件”。

31.7 一个放置案例

假设一个 LLM 推理服务需要 tensor_parallel_size=8。你有两种放置方式：

放置方式 看起来 实际影响
单机 8 卡，NVSwitch 8 张 GPU 都在同一高速互联域 TP 通信路径短，适合层内通信
4 台机器各 2 卡 也是 8 张 GPU 层内通信跨节点，decode 每步都可能被网络拖慢

对调度器来说，这两种都满足“8 张 GPU”。对 TP 来说，它们完全不是同一种资源。TP 把拓扑从硬件细节变成了应用正确性和性能要求。

31.8 平台需要表达 TP 组

TP 不是随便挑几张空 GPU。平台需要知道：

  • 哪些 GPU 属于同一个 TP group。
  • TP group 内 GPU 是否有高速互联。
  • TP group 是否跨 NUMA 或跨节点。
  • 推理副本是否需要独占整组 GPU。
  • 扩缩容时是否保持完整 TP group。
  • 监控指标是否按 group 聚合，而不是只看单卡。

这为后面 Kubernetes 调度打下基础：GPU 资源不能只用整数表达，还要表达拓扑和成组约束。

31.9 本章小结

Tensor Parallel 的一句话本质是：把同一层的 tensor 计算切给多张 GPU，用层内通信拼回结果。 它适合单层太大、单副本太慢或大模型推理需要多卡承载的场景。TP 的通信频率高，所以拓扑比 GPU 数量本身更重要。
