28  分布式 AI 的基本问题

28.1 本章要回答的问题

前四篇已经建立了单任务视角：训练如何读取数据、计算 loss、反向传播、更新 parameters；推理如何加载模型、处理请求、管理 KV Cache；GPU 如何通过 SM、HBM 和互联完成 tensor 计算。第五篇进入分布式。

分布式 AI 不是“把程序复制到多张 GPU 上”。它真正解决的是三个问题：

  • 算得太慢：单张 GPU 可以跑，但训练周期太长，或者推理吞吐不够。
  • 放不下：模型 weights、activation、gradients、optimizer state 或 KV Cache 超过单卡显存。
  • 服务扛不住：单个推理副本无法同时满足吞吐、延迟和长上下文需求。

分布式 AI 的本质，是把计算、显存状态和请求负载切到多个执行单元上，再用通信把它们重新拼成一个正确结果。

PyTorch 分布式体系把多进程、多设备通信抽象为 process group、rank、collective 等概念 (《PyTorch Distributed Overview》, 不详)。NCCL 则提供 AllReduce、AllGather、ReduceScatter 等 GPU 集合通信能力 (《NVIDIA NCCL User Guide》, 不详)。本篇不会先堆 API，而是先建立判断框架：切了什么，通信什么，省了什么，代价是什么。

Decision Map

分布式 AI 不是多用 GPU，而是选择切分维度

依据 PyTorch Distributed、NCCL collective 和常见大模型并行策略重绘。

先问瓶颈 算得慢、放不下、扛不住，分别对应不同切分方式。
Compute 切数据

Data Parallel / DDP：每张 GPU 一份模型，处理不同 batch shard。

通信：gradients AllReduce
Memory 切状态

FSDP / ZeRO：parameters、gradients、optimizer state 按 rank 分片。

通信：AllGather / ReduceScatter
Model 切模型

Tensor Parallel 切层内矩阵，Pipeline Parallel 切模型层。

通信：层内结果或 stage activation
Serving 切请求与阶段

多副本、TP group、prefill/decode 分离共同服务 SLA。

状态：KV Cache、队列、路由

28.2 先统一几个词

后面会持续使用这些术语，本书保持统一表达：

术语 含义 为什么重要
rank 分布式任务里的一个工作进程编号 日志、错误、通信、checkpoint 通常都按 rank 定位
world size 参与同一个分布式任务的总 rank 数 决定 global batch、通信规模和同步等待范围
replica 完整模型副本 Data Parallel / DDP 会在多张 GPU 上放多个 replica
shard 被切开的状态片段 FSDP / ZeRO 会把 parameters、gradients 或 optimizer state 切成 shard
collective 多个 rank 共同参与的通信操作 AllReduce、AllGather、ReduceScatter 都是 collective
topology GPU、CPU、网卡和节点之间的连接结构 同样 8 张 GPU，单机 NVSwitch 和跨节点以太网不是同一种资源

一个简单但非常有用的心智模型是：rank 是人，collective 是多人会议，topology 是他们之间的路。 人越多，不代表事情越快；会议越频繁，路越差，等待越多。

28.3 为什么分布式不会天然更快

单卡训练时，一个 step 大致是：

读数据 -> forward -> loss -> backward -> optimizer step -> checkpoint/metrics

多卡后，每一步都可能多出通信和同步：

每个 rank 读不同数据
每个 rank 做 forward/backward
多个 rank 同步 gradients 或状态 shard
每个 rank 更新或恢复自己的状态
所有 rank 在关键点等待一致

如果一个模型在单卡上每个 step 用 500 ms，多卡后不是简单除以 GPU 数。假设计算减少到 80 ms，但通信、等待和数据抖动合计 70 ms，那么 8 卡 step time 可能是 150 ms，而不是 62.5 ms。分布式优化的核心就是让“省下的计算或显存”大于“新增的通信和同步”。

28.4 四种切分方式

分布式 AI 常见策略可以按“切什么”来理解。

切分方式 切的对象 典型技术 解决的问题 主要代价
Data Parallel 数据 batch Data Parallel、DDP 算得慢，想缩短训练时间 gradients 同步，慢 rank 拖慢整体
State Sharding 训练状态 ZeRO、FSDP optimizer state、gradients、parameters 占显存 AllGather、ReduceScatter、checkpoint 更复杂
Tensor Parallel 单层 tensor 计算 Tensor Parallel 单层太大或推理单副本太慢 层内通信频繁，强依赖高速互联
Pipeline Parallel 模型层 Pipeline Parallel 模型太深、单卡放不下 micro-batch、pipeline bubble、stage 均衡
Service Parallel 请求和服务副本 多副本、prefill/decode 分离 在线吞吐、延迟、长上下文 路由、队列、KV Cache 状态管理

这张表是本篇的主线。后面每章都会沿着同一个问题展开：

  1. 它切了什么？
  2. 它为什么能省计算或显存？
  3. 它需要通信什么？
  4. 它在哪些场景有效？
  5. 它会给平台和调度带来什么新要求？

28.5 三类典型问题

28.5.1 算得太慢

如果单卡能放下模型，但训练太慢，优先考虑 Data Parallel / DDP。它让每张 GPU 处理不同 mini-batch，然后同步 gradients。直觉上像多个人同时批改不同试卷，最后一起讨论“模型参数应该往哪个方向改”。

这个策略最适合：

  • 模型可以完整放到每张 GPU。
  • 每个 step 的计算量足够大，能覆盖通信成本。
  • 数据读取能跟上多卡消耗。
  • 多个 rank 之间网络和拓扑稳定。

28.5.2 放不下

如果单卡放不下，就要先判断放不下的是哪一类状态：

显存对象 主要出现场景 常见解决方向
parameters 训练和推理都需要 FSDP、ZeRO-3、Tensor Parallel
gradients 训练 backward 产生 ZeRO-2/3、FSDP
optimizer state 训练更新 parameters 需要 ZeRO-1/2/3、FSDP
activation 训练 backward 需要中间结果 activation checkpointing、Pipeline Parallel
KV Cache LLM 推理保存上下文 PagedAttention、KV Cache 管理、分布式推理

这就是前面反复强调“显存账本”的原因。不了解显存主要被谁占用，就无法选择正确的分布式策略。

28.5.3 服务扛不住

推理分布式和训练分布式很不一样。训练通常围绕一个 global step 同步推进；推理面对的是不断进入和离开的在线请求。一次 LLM 请求还会分成 prefill 和 decode 两种计算形态，并持续消耗 KV Cache。

推理扩展要同时回答：

  • 是增加独立副本，还是把一个大模型切到多张 GPU？
  • 请求路由是否知道模型版本、KV Cache、上下文长度和队列水位？
  • batch 策略提升吞吐后，TTFT、TPOT 和 P99 是否仍满足 SLA？
  • 某个推理副本故障时，正在生成的流式请求如何处理？

28.6 平台视角：分布式任务不是一组普通进程

从应用代码看，分布式任务是一组 rank；从平台看，它至少有五类约束。

平台问题 为什么重要
同时启动 多数分布式训练要求所有 rank 都到齐，少一个 rank 可能整个任务无法开始
拓扑放置 Tensor Parallel 更适合放在高速 GPU 互联内，跨节点切层内矩阵通常代价很高
稳定通信 NCCL 超时、网络抖动、RDMA 配置错误会让训练卡住或失败
状态恢复 checkpoint 不只是 weights，还可能包含 optimizer state、scheduler、随机状态和 shard 元数据
rank 级观测 只看进程存活不够，要能看到每个 rank 的 step time、通信时间、显存和错误

这一章暂时不展开 Kubernetes。原因很简单：如果还没有理解 rank、shard、collective 和 topology，后面讨论 Pod、Job、Gang Scheduling、Device Plugin 和拓扑感知调度就会变成名词堆叠。先理解分布式 AI 的基本物理，再看平台如何表达这些约束。

28.7 常见误区

误区 更准确的说法
GPU 越多越快 GPU 增多会减少单卡计算，但也会增加通信、同步和排队
DDP 能解决显存不足 DDP 通常每张卡都有完整模型副本，不能从根本上减少单卡模型状态
FSDP/ZeRO 一定更快 它们主要为省显存而来，可能因为通信增加而变慢
Tensor Parallel 可以随便跨节点 TP 层内通信频繁，拓扑差会非常伤性能
推理扩副本就够了 大模型、长上下文、KV Cache 和 SLA 会让推理扩展变成状态管理问题

28.8 本章小结

分布式 AI 的核心不是“多机多卡”四个字，而是切分和重组。Data Parallel 切数据，FSDP/ZeRO 切训练状态，Tensor Parallel 切层内计算，Pipeline Parallel 切模型层，分布式推理切请求、模型副本和运行阶段。后面所有章节都围绕同一个标准展开：切了什么，通信什么，省了什么，代价是什么。
