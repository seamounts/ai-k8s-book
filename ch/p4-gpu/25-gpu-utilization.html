25  GPU 利用率到底是什么

25.1 本章要回答的问题

GPU 利用率是最常被看见的指标，也是最容易被误读的指标。

GPU utilization 只能说明某段时间 GPU 上有工作在跑，不能说明这些工作是否有效、是否高效、是否满足业务目标。

DCGM Exporter 基于 NVIDIA DCGM 暴露 GPU 指标给 Prometheus 等系统 (《NVIDIA DCGM Exporter Documentation》, 不详)。这些指标是平台观测的基础，但不能脱离训练日志和推理指标单独解释。

低利用率可能是数据慢、batch 小、同步多、请求少、通信等待。
高利用率可能是有效计算，也可能是低效 kernel、memory-bound、batching 过度。
正确做法训练看 step time 和收敛；推理看 TTFT/TPOT/P99；平台看队列和成本。

25.2 利用率低意味着什么

可能原因包括：

  • DataLoader 慢。
  • tokenizer 或 CPU 预处理慢。
  • 存储吞吐不足。
  • batch 太小。
  • kernel 太碎。
  • CPU-GPU 同步太频繁。
  • 多 GPU 等通信。
  • 推理请求太少或队列策略不合理。

25.3 利用率高也可能有问题

GPU 忙不代表业务好：

  • 低效 kernel 也能让 GPU 忙。
  • memory-bound operator 会占时间但吞吐不高。
  • 推理吞吐高但 P99 延迟不可接受。
  • batching 过度会让 GPU 忙，却让用户等待。
  • 训练 step 很快但 loss 不下降，说明有效进展不足。

25.4 应该组合哪些指标

场景 硬件指标 workload 指标
训练 SM 活跃度、HBM 使用、显存、NCCL 时间 step time、samples/sec、loss、DataLoader time
推理 显存、KV Cache、GPU 利用率 TTFT、TPOT、tokens/sec、P95/P99、队列长度
平台 GPU 空闲碎片、Pending、抢占 队列等待、租户配额、任务完成率

PyTorch Profiler 和 NVIDIA Nsight Systems 可以进一步帮助定位 CPU/GPU 时间线、operator 耗时和同步点 (《PyTorch Profiler》, 不详; 《NVIDIA Nsight Systems Documentation》, 不详)。

25.5 不同指标分别回答什么

指标 能回答 不能回答
GPU utilization GPU 这段时间是否有工作 工作是否有效、业务是否达标
memory used 显存占用多少 显存是否碎片、KV Cache 是否合理
SM active 计算单元是否忙 是否受 HBM 或同步限制
HBM bandwidth 显存带宽是否高 算法是否收敛、请求是否低延迟
power 功耗和热状态 workload 是否有有效进展
PCIe/NVLink throughput 数据移动是否多 数据移动是否必要

因此，指标解释一定要回到 workload：训练看 step 和 loss，推理看 TTFT/TPOT/P99，平台看队列、公平性和成本。

25.6 一个判断流程

看到 GPU utilization 低，按顺序问：

  1. 当前是否有足够工作进入系统？训练数据是否供给稳定，推理请求是否足够？
  2. CPU 是否在做 tokenizer、DataLoader、预处理或 kernel launch？
  3. 是否有 CPU-GPU 同步、checkpoint、日志或远端存储阻塞？
  4. 是否在等 NCCL 或跨节点通信？
  5. batch shape 是否太小或太碎？

看到 GPU utilization 高，也要问：

  1. samples/sec 或 tokens/sec 是否真的高？
  2. TTFT/P99 是否变差？
  3. HBM 带宽是否打满，说明 memory-bound？
  4. 是否有大量小 kernel 或低效 operator？
  5. 训练 loss 是否有有效下降？

25.7 本章小结

  • GPU utilization 是入口指标，不是结论。
  • 训练看有效 step 和收敛，推理看延迟和吞吐，平台看公平和成本。
  • 下一章把这些指标用于训练瓶颈定位。
