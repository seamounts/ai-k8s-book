22  SM、Tensor Core 与 kernel

22.1 本章要回答的问题

第二章只预告了 kernel，没有强行让读者理解 GPU 构造。本章正式解释：

kernel 是被发到 GPU 上执行的函数；GPU 会把它组织成 grid、block、thread，并在 SM 上以 warp 为单位调度。

CUDA 编程指南说明，CUDA kernel 会由许多 thread 并行执行，这些 thread 按 block 和 grid 组织；硬件执行时，thread 会按 warp 分组调度 (《CUDA C++ Programming Guide》, 不详)。

22.2 operator 与 kernel 的关系

用户写的是模型代码。框架看到的是 operator。GPU 真正执行的是这些 operator 对应的 kernel 或 kernel 组合。

model code -> operator -> backend library -> kernel -> SM/Tensor Core/HBM

比如一个线性层看起来是 Y = XW + b，底层可能调用 cuBLAS 矩阵乘法 kernel，再加上 bias、activation 或融合后的 kernel。FlashAttention 这类优化，本质上也是重新设计 kernel 和内存访问方式，减少中间数据搬运。

一个 operator 不一定只对应一个 kernel。它可能：

情况 例子 性能影响
一个 operator 一个 kernel 大矩阵乘法 启动开销相对小，计算成块
一个 operator 多个 kernel 复杂 attention 或 normalization kernel 之间可能读写中间结果
多个 operator 融合成一个 kernel bias + activation + dropout 减少 HBM 读写和 kernel launch
走库实现 cuBLAS、cuDNN、FlashAttention 通常比手写普通实现更高效

平台工程师不写 CUDA，也要知道这个层次关系。因为 profiler 里看到的通常不是“模型层”，而是 kernel、operator 和 runtime 调用。

22.3 SM 与 warp

SM 是 NVIDIA GPU 上的核心执行单元。warp 通常是一组 32 个 thread。可以把 warp 理解成同一节拍推进的一队 thread：

  • 如果它们执行同一路径，效率高。
  • 如果发生分支发散，效率下降。
  • 如果访问连续内存，吞吐更好。
  • 如果访问分散内存，带宽效率变差。

平台工程师不一定写 CUDA，但必须理解：GPU 喜欢规则、成批、连续、可并行的工作。

CUDA Best Practices Guide 强调内存访问模式会影响性能，连续、合并的访问通常更容易获得高带宽 (《CUDA C++ Best Practices Guide》, 不详)。这解释了为什么“shape 看起来只是数学维度”，实际却会改变 kernel 选择、访存方式和吞吐。

22.4 Tensor Core 加速什么

Tensor Core 面向矩阵块乘加。NVIDIA 文档说明，Tensor Core 可加速 matrix multiply-and-accumulate，常见模式包括较低精度输入和较高精度累加 (《GPU Performance Background User’s Guide》, 不详)。混合精度训练文档也强调 FP16/FP32、loss scaling 和 Tensor Core 对 shape 的要求 (《Train With Mixed Precision》, 不详)。

因此，Tensor Core 的收益取决于：

  • operator 是否能映射成矩阵块计算。
  • dtype 是否合适，例如 FP16、BF16、TF32、INT8。
  • tensor shape 是否匹配高效路径。
  • 框架和后端库是否启用相应实现。

Tensor Core 不会加速数据加载、请求排队、网络通信或 checkpoint 写入。

22.5 kernel launch 和小算子问题

GPU 很强，但发起一次 kernel 也有开销。如果模型或数据处理产生大量很小的 operator，GPU 可能频繁启动 kernel，却每次只做很少计算。表现是：

  • GPU 利用率不稳定。
  • CPU 时间线里 kernel launch 很密集。
  • profiler 里出现大量小 kernel。
  • 融合后性能明显提升。

这就是算子融合、编译优化和图优化的价值所在。它们不是“让代码更炫”，而是减少小算子和中间数据搬运，让 GPU 做更大块、更连续的工作。

22.6 occupancy 不是唯一目标

很多资料会提到 occupancy，即 SM 上活跃 warp 的程度。occupancy 太低可能表示并行度不足，但 occupancy 高也不保证性能好。如果 kernel 受 HBM 带宽限制，更多 warp 也未必能提高吞吐；如果访存不连续，高 occupancy 也可能只是更多线程一起等内存。

所以判断 GPU kernel 性能要同时看：

指标 回答什么
SM 活跃度 计算单元是否有工作
Tensor Core 使用 矩阵计算是否走加速路径
HBM 带宽 是否受显存带宽限制
kernel launch 数量 是否小算子过多
CPU-GPU 同步 是否被同步点打断

22.7 本章小结

  • operator 是框架层抽象，kernel 是 GPU 侧执行函数。
  • thread/block/grid 是 CUDA 的组织方式，warp/SM 是硬件执行视角。
  • Tensor Core 加速矩阵块乘加，不是加速整个系统。
  • 下一章讲显存层级：为什么算力再强，也会被 HBM、缓存和数据搬运限制。
