21  GPU 与 CPU 的分工

21.1 本章要回答的问题

前面已经讲过训练 step 和推理 request。现在进入第四篇，先回答最基础的问题：GPU 为什么适合 AI，CPU 又为什么仍然重要？

GPU 不是更快的 CPU。CPU 擅长控制和通用逻辑，GPU 擅长把同一类 tensor 计算铺到大量数据上并行执行。

NVIDIA GPU 性能背景资料把现代 GPU 描述为由多个 SM、L2 Cache 和高带宽 DRAM 组成的并行处理器 (《GPU Performance Background User’s Guide》, 不详)。这句话的工程含义是：GPU 性能不是单点算力，而是执行单元、缓存、显存带宽和数据供给共同作用。

Result返回与控制loss、gradient、tokens、metrics、checkpoint 再回到控制逻辑。

GPU 不是更快的 CPU。CPU 慢、同步多、数据送不及时，都会让 GPU 等。

21.2 CPU 负责什么

在 AI workload 里，CPU 常常负责：

  • Python 进程和框架调度。
  • DataLoader、tokenizer、图像解码和数据增强。
  • operator 发起和 CPU-GPU 同步。
  • 网络、存储、日志、checkpoint 和控制逻辑。
  • 推理服务里的 router、queue、HTTP/gRPC 处理。

所以 GPU 利用率低时，不要只盯 GPU。CPU 数据处理、tokenizer、对象存储、小文件、同步点，都可能让 GPU 等。

CPU 还决定了“GPU 是否能连续工作”。例如训练时 DataLoader worker 太少，GPU 会算一阵等一阵；推理时 tokenizer 或请求队列在 CPU 上堵住，GPU 可能明明空着，用户却已经在等待首 token。

21.3 GPU 负责什么

GPU 负责核心 tensor 计算：

  • 矩阵乘法。
  • convolution。
  • attention。
  • normalization。
  • embedding lookup 相关操作。
  • 大量 elementwise operator。

GPU 擅长的不是“理解文本”，而是规则、成批、可并行的数字计算。batch、tensor shape、dtype 和 operator 实现，会决定 GPU 是否真的被喂满。

GPU 最喜欢三类工作：

特征 为什么适合 GPU
大块矩阵计算 数据复用高，能使用 Tensor Core
规则的 shape kernel 更容易走高效实现
连续的内存访问 更容易用满显存带宽

GPU 不喜欢小而碎、频繁同步、分支复杂、访存分散的工作。这就是为什么同样的模型，batch、dtype、shape、算子融合和数据布局不同，性能会差很多。

21.4 Host 与 device

CUDA 编程模型区分 host 和 device：host 通常是 CPU 侧程序，device 是 GPU 侧执行环境；host 发起 kernel，device 执行 kernel (《CUDA C++ Programming Guide》, 不详)。

这解释了一个常见性能现象：

现象 可能原因
GPU 间歇性空闲 CPU 数据加载慢，或 host/device 同步太频繁
小模型推理 GPU 利用率低 单次请求计算量太小，batch 不够
训练 step 抖动 DataLoader、checkpoint、通信或同步点抖动
GPU 忙但吞吐不高 operator 低效、显存带宽受限或 shape 不合适

21.5 CPU-GPU 协作的时间线

一次训练 step 或推理请求，大致会这样推进：

  1. CPU 读取数据、执行 tokenizer 或预处理。
  2. CPU 把输入 tensor 准备好，必要时从 host memory 传到 GPU memory。
  3. CPU 调用框架后端，发起一个或多个 GPU kernel。
  4. GPU 执行 kernel，读写 HBM、L2、寄存器和 shared memory。
  5. 如果需要结果、日志、同步或下一步控制，CPU 和 GPU 会发生同步。

性能问题经常藏在第 2 步和第 5 步：数据搬运太慢，或者同步点太多。平台排障时要看端到端时间线，而不是只看 GPU 某一个瞬间忙不忙。

21.6 平台含义

Kubernetes 里给 Pod 分配 GPU，不等于给了完整 AI 性能。还要同时考虑：

资源 为什么影响 GPU 效率
CPU cores DataLoader、tokenizer、runtime 控制面需要 CPU
host memory 数据预取、pinned memory、缓存会消耗内存
local disk / cache 模型加载和数据读取会影响冷启动和吞吐
NUMA / PCIe 拓扑 CPU、GPU、网卡位置影响数据搬运和通信
网络 分布式训练、远端存储、模型拉取都依赖网络

所以后面讲调度时，不能只说“这个任务申请 8 张 GPU”。更准确的问题是：这 8 张 GPU 配了多少 CPU、内存、存储、网络，以及它们在拓扑上是否适合这个任务。

21.7 本章小结

  • CPU 和 GPU 是协作关系，不是替代关系。
  • CPU 慢会让 GPU 空等，GPU 忙也不代表端到端效率高。
  • 平台排障要沿着数据、CPU、GPU、显存、通信和控制路径一起看。
