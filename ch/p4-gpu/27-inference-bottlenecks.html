27  推理性能瓶颈分析

27.1 本章要回答的问题

推理慢，也不一定是 GPU 算得慢。LLM 推理尤其复杂，因为请求会排队，prefill 和 decode 特性不同，KV Cache 会占显存，batching 会改变延迟和吞吐。

27.2 按请求路径拆瓶颈

阶段 典型瓶颈 指标
router/queue 排队过长、优先级不合理 queue time、timeout
tokenizer CPU 开销 tokenizer time、CPU
prefill prompt 长、batch 大 TTFT、prefill time
KV Cache 显存不足、碎片、长上下文 KV Cache 使用、OOM
decode 每 token 调度、显存带宽 TPOT、tokens/sec
streaming 连接、网关、客户端慢 stream error、断连

27.3 TTFT 与 TPOT 分开看

TTFT 高，优先看：

  • queue time。
  • tokenizer。
  • prefill。
  • 冷启动和模型加载。
  • batching 等待时间。

TPOT 高，优先看：

  • decode batch 调度。
  • KV Cache 访问。
  • 显存带宽。
  • tensor parallel 通信。
  • runtime kernel 实现。

27.4 TTFT 和 TPOT 的优化方向不同

指标变差 常见原因 优化方向
TTFT 高 queue delay、冷启动、tokenizer、长 prompt、prefill 拥塞 减少排队、预热副本、分离长 prompt、优化 prefill
TPOT 高 decode batch 大、KV Cache 访问慢、HBM 带宽、跨卡通信 调整 continuous batching、优化 cache、减少并发、改 runtime
P99 高 长短请求混部、租户干扰、cache 压力、GC/重启 分桶、优先级、配额、隔离副本池
tokens/sec 低 batch 不足、小请求、GPU 空等 动态 batching、请求聚合、提高并发
OOM KV Cache、长上下文、max tokens 过大 准入控制、上下文配额、cache 回收

27.5 batching 的双刃剑

Triton dynamic batching 文档展示了一个普遍原则：等待一小段时间可以合并请求并提高吞吐，但等待本身会增加延迟 (《Triton Inference Server: Dynamic Batcher》, 不详)。

LLM 的 continuous batching 更进一步：请求会在 decode 过程中不断进入和退出 batch。它提高 GPU 利用率，但也让调度、显存和公平性更复杂。

27.6 三个推理性能案例

案例一：TTFT 高，但 TPOT 正常。
通常说明首 token 前的路径有问题：queue、tokenizer、模型冷启动或 prefill。优先看 queue time、prefill time、prompt 长度分布和副本 ready 状态。

案例二：TTFT 正常，但 TPOT 变慢。
通常说明 decode 阶段有问题：KV Cache、decode batch、显存带宽或 tensor parallel 通信。优先看 active sequences、KV Cache 使用率、tokens/sec 和 decode kernel。

案例三：平均延迟正常，P99 很差。
通常说明长尾请求或多租户干扰。优先看 prompt/output 长度分布、租户维度指标、长请求是否和短请求混在同一队列。

27.7 平台落点

推理性能优化最后会落到平台能力：

能力 解决什么
模型预热和 ready gate 冷启动和 TTFT 抖动
请求准入和配额 KV Cache OOM 和长上下文冲击
队列优先级和分桶 长短请求混部、租户干扰
runtime 指标暴露 TTFT、TPOT、cache、batch 观测
弹性伸缩 高峰期容量和成本平衡
多副本/多池路由 不同 SLA、不同模型、不同上下文长度隔离

27.8 本章小结

推理优化不能只追求 GPU 忙。交互式 LLM 服务首先要守住 TTFT 和 P95/P99，再在这个边界内提高 tokens/sec 和 GPU 利用率。
