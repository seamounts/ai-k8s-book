30  从 Kubernetes 视角重新理解 AI 任务

30.1 本章要回答的问题

前几章从 AI 工作负载本身出发，解释了计算、显存、数据、通信、训练、推理和调度难点。现在要把这些问题放回 Kubernetes。

Kubernetes 不直接理解“训练”“推理”“梯度同步”“KV Cache”这些概念。它理解的是 API 对象、控制器、调度器、Pod、Service、Volume 和指标。

本章给出第七篇的开篇判断：

AI on Kubernetes 的核心，是把 AI 任务的生命周期和资源约束，翻译成 Kubernetes 可以声明、控制、调度、观测和治理的对象。

30.2 Kubernetes 看到什么

Kubernetes 的基本工作方式是声明式控制。用户提交期望状态，控制器不断观察实际状态，并尝试让实际状态接近期望状态。

对 AI 平台来说，最重要的对象包括：

Kubernetes 对象 在 AI 场景中的作用
Pod 训练 worker、推理副本、Ray worker、数据处理进程
Job 一次性训练、批处理、离线任务
Deployment 在线推理服务副本
StatefulSet 需要稳定身份或存储的服务组件
Service 推理服务发现、worker 互相发现、入口抽象
PersistentVolumeClaim 数据集、checkpoint、模型权重、缓存
ConfigMap / Secret 配置、启动参数、凭证
CustomResourceDefinition PyTorchJob、TFJob、VolcanoJob、RayJob、InferenceService 等扩展 API
Scheduler 把 Pod 放到合适节点
Controller / Operator 把高层任务对象转化为底层 Pod、Service、PVC 等资源

这些对象不是孤立工具。它们共同构成 AI 任务的控制回路。

30.3 AI 任务的 Kubernetes 控制回路

这个控制回路说明了 AI on Kubernetes 的分层：

  • CRD 表达高层任务意图。
  • Operator 把任务意图转成 Kubernetes 资源。
  • Scheduler 负责资源分配和节点选择。
  • Runtime 在容器内执行训练或推理。
  • Metrics 和 Events 把运行状态反馈给平台。

如果其中任何一层缺失，平台就会退化成“提交 YAML 的脚本集合”。

30.4 Pod：最小运行单位，但不是任务边界

Pod 是 Kubernetes 调度和运行的基本单位。官方文档把 Pod 描述为一组共享运行上下文的容器；实际使用中，单容器 Pod 最常见。

AI 任务里，一个 Pod 可能是：

  • 一个单卡训练进程。
  • 一个分布式训练 worker。
  • 一个 parameter server。
  • 一个 Ray worker。
  • 一个 vLLM 推理副本。
  • 一个 Triton model server。

但 Pod 通常不是 AI 任务的完整边界。一个分布式训练任务可能包含多个 worker Pod；一个推理服务可能包含多个模型副本、router 和 autoscaler；一个 Ray 应用可能包含 head Pod 和多个 worker Pod。

所以，平台不能只管理 Pod，还要管理 Pod 之间的关系。

30.5 Job：一次性任务的基础

Kubernetes Job 的语义是运行一个任务直到完成。官方 Job 控制器会创建 Pod，并在 Pod 失败或被删除时根据策略创建新的 Pod。

这很适合一类简单 AI 任务：

  • 单机单卡训练。
  • 离线 batch 推理。
  • 数据预处理。
  • 模型转换。
  • 小规模实验任务。

但普通 Job 对复杂分布式训练不够。它不天然表达 master/worker 角色、rank、world size、最小可运行成员、训练框架启动方式或队列公平性。

这就是 PyTorchJob、TFJob、MPIJob、VolcanoJob、RayJob 等 CRD 出现的原因。它们不是为了替代 Kubernetes，而是把 AI 任务的高层语义补进 Kubernetes API。

30.6 CRD 与 Operator：把专业知识放进控制面

CustomResourceDefinition 允许 Kubernetes 扩展新的 API 类型。Operator pattern 则把自定义资源和控制器结合起来，用控制器自动执行领域知识。

在 AI 场景中，Operator 的价值是把“人知道但 Kubernetes 原生对象不知道”的知识自动化。

例如：

  • PyTorchJob Operator 知道如何创建 master 和 worker。
  • TFJob Operator 知道 TensorFlow 的 chief、worker、PS 等角色。
  • Ray Operator 知道如何创建 Ray head 和 Ray worker。
  • KServe Controller 知道如何把 InferenceService 转成模型服务相关资源。
  • Volcano Controller 和 Scheduler 知道如何处理 VolcanoJob、PodGroup、Queue 等批任务语义。

这就是 AI 平台从“脚本提交系统”走向“声明式平台”的关键。

30.7 Service：推理和分布式协同的入口

Service 在 Kubernetes 中提供稳定的服务发现和访问入口。对推理服务来说，它可以把请求路由到一组模型副本。对分布式训练来说，它也可能用于 worker 发现、master 地址暴露或控制面通信。

但 Service 不解决所有推理问题。它解决的是网络入口和服务发现，不解决：

  • 模型加载。
  • KV Cache 管理。
  • 请求动态 batching。
  • token streaming。
  • 灰度发布。
  • 模型版本治理。
  • GPU 显存调度。

这些能力需要由推理运行时、模型服务平台、网关、控制器和观测系统共同完成。

30.8 Volume：数据、checkpoint 和模型权重

AI 任务经常离不开持久化存储。

训练需要读取数据集、写 checkpoint、保存日志和模型产物。推理需要加载模型权重，有时还需要缓存或挂载模型仓库。Kubernetes 通过 Volume、PersistentVolume 和 PersistentVolumeClaim 抽象存储。

存储的关键不是“能不能挂载”，而是：

  • 吞吐能否喂饱 GPU。
  • 多个 worker 并发读取时是否成为瓶颈。
  • checkpoint 写入是否影响训练 step。
  • 模型加载是否导致推理冷启动过长。
  • 本地缓存和远端对象存储如何配合。

存储慢，GPU 会空等；checkpoint 慢，训练会抖动；模型加载慢，推理扩容会变慢。

30.9 Scheduler：资源分配入口

Kubernetes 默认调度器负责把未绑定节点的 Pod 分配到 Node。它会根据资源、约束和打分策略选择节点。

AI 平台会在这个基础上叠加更多语义：

  • 通过节点标签表达 GPU 型号、拓扑、网络能力。
  • 通过 taint/toleration 隔离 GPU 节点池。
  • 通过 resource request 申请 GPU。
  • 通过自定义调度器或批调度器实现队列、gang 和公平性。
  • 通过 metrics 反馈真实运行状态。

关键是：Scheduler 看到的是 Pod 级别的调度输入。如果训练或推理的高层语义没有被 CRD、Operator、队列系统或调度插件表达出来，调度器就无法凭空知道。

30.10 AI 平台的分层模型

可以把 AI on Kubernetes 平台分成六层：

层次 核心问题 典型组件
用户入口层 用户如何提交训练和推理 Portal、CLI、API、Notebook
任务抽象层 如何表达 AI 任务 PyTorchJob、TFJob、RayJob、InferenceService
控制器层 如何创建和维护底层资源 Operator、Controller
调度层 如何分配 GPU 和队列资源 kube-scheduler、Volcano、Kueue、KAI Scheduler
运行时层 如何执行训练和推理 PyTorch、TensorFlow、Ray、vLLM、Triton
观测治理层 如何看见、恢复和优化 Prometheus、DCGM、日志、事件、Profiler

这张表是后面全书的地图。看到任何一个新技术，先问它在哪一层，再问它解决哪一层的问题。

30.11 常见误区

误区一：有了 Kubernetes，就自然有了 AI 平台。
Kubernetes 提供通用控制面，但 AI 平台还需要任务抽象、训练/推理运行时、GPU 调度、模型治理和观测能力。

误区二：CRD 越多，平台越强。
CRD 的价值在于表达真实任务语义。如果只是换一种 YAML 写法，没有配套控制器、状态管理和调度语义，复杂度会高于收益。

误区三：Service 就等于模型服务平台。
Service 只解决网络发现和访问入口。模型服务还需要模型加载、版本、运行时、扩缩容、灰度和指标。

误区四：调度器可以自动理解训练任务。
调度器只能根据输入做决策。训练任务的 gang、rank、拓扑、队列和恢复语义必须被显式表达。

30.12 本章小结

读完本章，需要形成四个判断：

  • Kubernetes 是 AI 平台的承载层，不是 AI 平台本身。
  • Pod 是最小运行单位，但 AI 任务往往需要更高层的 CRD 和 Operator 表达。
  • Scheduler 只能根据已表达的资源和约束做决策，不能自动理解训练或推理语义。
  • 生产级 AI 平台必须把任务抽象、控制器、调度、运行时和观测治理连成闭环。

到这里，我们开始把前面几篇建立的 AI workload、训练流程、推理流程、GPU 性能和平台化问题映射到 Kubernetes。后续章节会继续展开 CRD、Operator、训练任务架构、推理服务架构、队列、调度器和资源治理。
