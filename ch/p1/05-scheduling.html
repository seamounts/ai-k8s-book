39  为什么 AI 任务难调度

39.1 本章要回答的问题

如果只看 Kubernetes API，AI 任务也只是 Pod、Job、Deployment 或自定义资源。既然 Kubernetes 已经有调度器，为什么还需要 Volcano、Kueue、KAI Scheduler、Gang Scheduling、Fair Share、DRF、Backfilling 和拓扑感知？

答案很直接：

AI 任务难调度，不是因为 Pod 难放置，而是因为 GPU 稀缺、任务强协同、资源拓扑敏感、运行时间长、失败成本高，并且多租户之间存在真实的公平性冲突。

本章不先讲具体调度器，而是先讲这些调度能力背后的系统压力。

39.2 普通调度和 AI 调度的差别

Kubernetes 默认调度器的核心工作，是为尚未绑定节点的 Pod 选择一个合适的 Node。它会过滤不可行节点，再对可行节点打分，最后完成绑定。

这个模型非常适合大量无状态服务，也能处理很多普通批任务。但 AI 任务经常超出“单个 Pod 找一个节点”的简单模型。

问题 普通服务 AI 训练 / 推理
资源粒度 CPU、内存为主 GPU、显存、拓扑、网络、存储
任务形态 多副本可独立运行 分布式 worker 可能强协同
启动要求 副本可以逐步可用 训练常要求一组 Pod 同时可用
运行时长 长短都有，常可滚动替换 训练长周期，失败代价高
拓扑敏感度 通常较低 多 GPU / 多节点任务很敏感
公平性 服务容量治理 多租户 GPU 配额和队列治理

因此，AI 调度不是否定 Kubernetes 默认调度，而是在 AI 场景中补足它没有表达或没有优化的任务语义。

39.3 难点一：GPU 是稀缺且不均匀的资源

CPU 和内存通常可以细粒度切分。GPU 则不同。

一张 GPU 可能被整个训练任务独占，也可能通过 MIG、time-slicing、MPS 等方式共享。不同型号 GPU 的显存、算力、互联能力不同。即使同型号 GPU，所在节点、PCIe 拓扑、NVLink/NVSwitch 拓扑、网卡位置也可能影响性能。

这会带来两个直接问题：

  • 资源碎片：集群里还有 8 张空闲 GPU，但分散在 8 个节点上，无法满足一个需要单机 8 卡的任务。
  • 性能不确定：任务拿到了足够数量的 GPU，但拓扑较差，导致通信慢，训练效率低。

所以，AI 调度不能只看“剩余 GPU 数量”，还要看资源形态。

39.4 难点二：分布式训练需要成组调度

很多分布式训练任务不是“有几个 Pod 先跑几个”。它需要一组 worker 同时进入有效运行状态。

如果一个任务需要 8 个 worker，但调度器只放进去了 4 个，结果可能是：

  • 已启动 worker 等待其他 worker。
  • GPU 被占住但没有有效训练。
  • 其他任务也拿不到这些被占住的 GPU。
  • 集群整体利用率下降。

这就是 Gang Scheduling 的来源。它的核心思想不是“更高级的调度”，而是一个简单约束：

一组强协同任务，要么满足最小运行成员数后一起运行，要么暂时都不运行。

VolcanoJob 的 minAvailable、PodGroup 的最小成员约束，以及类似调度系统中的 gang 语义，都是在表达这个问题。

39.5 难点三：长任务的失败成本高

训练任务经常运行很久。失败一次，损失的不是一次请求，而是大量已经消耗的 GPU 时间。

所以调度器和平台必须关心：

  • 任务是否支持 checkpoint。
  • 抢占后能否恢复。
  • 恢复需要多长时间。
  • 失败是单个 Pod 问题，还是整个 job 需要重启。
  • 是否应该优先保护已经运行很久的任务。

这也是 AI 调度和普通在线服务调度的重要差异。在线服务通常通过副本冗余和请求重试恢复；训练任务则需要 checkpoint、resume 和 job 级状态管理。

39.6 难点四：多租户公平性不是平均分

GPU 集群通常服务多个团队、项目或业务线。每个租户都希望任务尽快运行，但 GPU 不可能无限供给。

公平性并不等于简单平均分。真实集群里常见需求包括：

  • 重要队列拥有更高权重。
  • 某些队列有最小保障资源。
  • 某些队列可以借用空闲资源。
  • 借用资源在高优任务到来时可以被回收。
  • 不同租户可能使用不同 GPU 型号。

Fair Share、DRF、Quota、Priority、Preemption 这些机制，都是为了处理多租户资源冲突。

其中 DRF 的直觉是：不要只看某一种资源，而要看每个租户在“主导资源”上的占用比例。例如某个租户主要占 GPU，另一个租户主要占 CPU 和内存，公平性就不能只按 Pod 数量计算。

39.7 难点五：利用率和公平性会冲突

调度系统经常同时追求两个目标：

  • 公平：不同租户按规则获得资源。
  • 利用率：GPU 尽量不要空闲。

这两个目标会冲突。

假设一个大训练任务排在队首，需要 16 张 GPU，但当前只有 8 张空闲 GPU。如果严格按队列顺序等待，8 张 GPU 会闲置。如果允许一个只需要 2 张 GPU 的短任务先运行，就能提高利用率，但可能影响队首任务等待时间。

Backfilling 就是在处理这个矛盾：在不破坏关键公平性和预约约束的前提下，把短任务填进资源空档。

这说明调度策略不是越“公平”越好，也不是利用率越高越好。调度器必须在公平、吞吐、等待时间、抢占成本和业务优先级之间做权衡。

39.8 难点六：推理也有调度问题

很多人把 AI 调度只理解成训练调度。实际上，推理也有调度问题，只是形态不同。

推理调度至少包括：

  • 请求路由到哪个模型副本。
  • 新副本放到哪个节点。
  • 是否使用 GPU 共享。
  • 是否按模型大小、显存余量和 KV Cache 状态做路由。
  • 扩容时如何避免冷启动冲击。
  • 多模型服务如何避免互相干扰。

大模型推理如果使用 tensor parallel 或 pipeline parallel，也会重新引入多 Pod / 多 GPU 协同问题。PD disaggregation、prefill/decode 分离、多节点推理等架构，会让推理调度越来越接近复杂的分布式系统调度。

39.9 Kubernetes 落地视角

Kubernetes 的默认调度器负责 Pod 到 Node 的绑定。它是 AI 平台的基础，但不是全部。

AI 平台通常会在它之上或旁边引入额外组件：

能力 典型实现
队列和准入 Kueue、Volcano Queue、平台自研队列
Gang Scheduling Volcano PodGroup、VolcanoJob、其他批调度器
公平性 Fair Share、DRF、Quota、Priority
拓扑感知 节点标签、拓扑插件、NVIDIA/NIC/GPU 拓扑信息
GPU 共享 MIG、time-slicing、MPS、vGPU
任务控制 Kubeflow Training Operator、VolcanoJob、RayJob、KServe
观测反馈 DCGM、Prometheus、训练日志、推理指标

这些组件的共同目标，是把 AI 工作负载的真实约束传递给调度和控制系统。

39.10 常见误区

误区一：调度就是把 Pod 放到节点。
这是 Kubernetes 调度的表层动作。AI 调度还要处理队列、最小成员数、拓扑、公平性、抢占和恢复成本。

误区二：GPU 空闲就是调度器没做好。
不一定。可能是队首大任务等待完整资源，也可能是资源碎片、拓扑约束或配额策略造成的结果。

误区三：抢占一定能提高效率。
抢占会释放资源，但也会浪费被抢占任务的已完成计算。没有 checkpoint 的长训练任务被抢占，成本可能很高。

误区四：推理服务只需要 HPA。
HPA 能处理部分扩缩容问题，但 LLM 推理还需要考虑模型加载、显存、KV Cache、冷启动、batching 和路由策略。

39.11 本章小结

读完本章，需要形成四个判断：

  • AI 调度难在任务语义，而不是 Pod API。
  • 分布式训练需要成组调度，否则可能占用 GPU 却没有有效计算。
  • 多租户 GPU 集群必须在公平性、利用率和抢占成本之间权衡。
  • 推理服务也有调度问题，尤其是大模型推理和多模型服务。

后面讲 Volcano、Kueue、KAI Scheduler 和推理调度时，本章就是它们的系统背景。
