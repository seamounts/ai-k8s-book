1  AI 系统到底在做什么

1.1 本章要回答的问题

很多平台工程师第一次接触 AI 任务时，会先看到容器、镜像、GPU、任务配置和各种框架名词。这样学习很容易走偏：还没有理解 AI 系统本身，就开始讨论如何把它放进平台。

本书不从 Kubernetes 开始，也不从某个框架开始。本章先回答一个更基础的问题：

AI 系统到底在做什么？

一针见血地说：AI 系统是在用数据驱动一个可计算函数：训练时调整这个函数的参数，推理时用已经学到的参数产生结果。

这不是一句比喻，而是工业界通用的工程视角。Google 的机器学习术语表把 training 定义为确定模型理想 weights 的过程，把 inference 定义为使用已学习的 weights 进行 prediction 的过程 (《Machine Learning Glossary: ML Fundamentals》, 不详)。PyTorch 教程也明确说，tensor 用来编码模型的 input、output 和 parameters，并且可以运行在 GPU 等加速器上 (《Tensors》, 不详)。

所以，本书前几章会一直坚持一个主线：

Core Spine

全书主线：从 data 到 result

先看懂这条线，再看 GPU、分布式、Kubernetes 和调度。

Data 真实输入

图片、文本、日志、样本、请求。系统首先要把它们变成可计算的数字。

Tensor 有形状的数字

tensor 有 shape、dtype 和 device。GPU 最终处理的是这些数字结构。

Parameters 模型学到的规则

parameters 是模型可更新或可加载的数值状态，通常占用大量显存。

Compute operator 执行

矩阵乘法、attention、归一化、激活函数等 operator 组成计算路径。

Result 输出结果

分类、文本、向量、分数或下一步 token，最终服务于训练或推理目标。

Training：用 result 修正 parameters result 与 label 计算 loss，backward 产生 gradient，optimizer 更新 parameters。
Inference：用 parameters 生成 result parameters 通常不变，请求进入服务，runtime 读取 weights 并尽快返回结果。

这张图就是全书的地基。第二篇训练流程，会沿着这条线解释 result 如何变成 loss、gradient 和新的 parameters；第三篇推理流程，会沿着这条线解释请求如何经过 weights、KV Cache 和 batching 变成流式输出；第四篇以后，GPU、分布式和 Kubernetes 都是在回答：怎样让这条线在集群里跑得更快、更稳、更可控。

术语也从这里统一：正文中，tensor、batch、activation、gradient、operator、kernel、KV Cache 这类专有名词第一次成为核心概念时给中文解释，后文尽量使用同一个英文专名，不在“中文名”和“英文名”之间来回切换。章节标题为了中文阅读体验，可能保留中文表达。

只要抓住这条主线，后面学习 GPU、分布式训练、推理服务、调度、Kubernetes 和各种框架时，就不会被名词牵着走。

1.2 从一个最小例子开始

先看一个最小的图片分类任务。用户给系统一张图片，系统输出“这张图属于哪个类别”。

从外面看，它像一个普通程序：输入图片，返回结果。
从系统内部看，它至少经过五步：

  1. 把图片读取进来。
  2. 把图片转换成数字数组，也就是 tensor。
  3. 模型用 parameters 对这个 tensor 做一串计算。
  4. 得到每个类别的分数，通常叫 logits。
  5. 把 logits 转成更容易解释的概率或排序，选择分数最高的类别作为结果。

TensorFlow 的基础图片分类教程就是用这个路径讲清楚一个最小模型：输入图片，训练网络，得到分类结果，并用 softmax 把线性输出转换成概率 (《Basic classification: Classify images of clothing》, 不详)。这类教程之所以经典，是因为它把 AI 系统最核心的几件事都暴露出来了：数据、tensor、model、parameters、compute、result。

这里最重要的不是“图片分类”这个任务，而是这个结构：输入数据、tensor、model parameters、compute、result。几乎所有训练和推理系统，都可以从这个结构展开。

再往工程里看，训练和推理只是这条路径的两种运行方式：

  • 训练会把 result 和真实标签比较，得到 loss，再通过 gradient 更新 parameters。
  • 推理不会更新 parameters，而是尽快把 result 返回给用户或下游系统。

因此，平台工程师理解 AI 任务，第一步不是看它用了哪个框架，而是先问：数据在哪里，tensor 形状是什么，parameters 有多大，compute 放在哪里执行，result 如何被使用。

1.3 模型是什么

模型不是一个会思考的实体。对系统工程来说，模型首先是一组 parameters（参数）。

parameters 可以理解成模型已经学到的数字规则。在线性模型里，parameters 可能只是几个 weights；在大语言模型里，parameters 可能是数十亿、数百亿甚至更多的矩阵数值。模型越大，parameters 越多，占用的显存和加载时间也越大。

因此，平台工程里谈“模型”，至少要同时看到三件事：

视角 关心什么 系统问题
算法视角 模型能不能学到规律 准确率、泛化能力、收敛速度
计算视角 模型要执行多少 tensor 计算 GPU 算力、底层执行效率、吞吐
系统视角 模型参数如何存放、加载、更新和服务化 显存、存储、网络、部署和恢复

本书主要站在第三个视角，但不会跳过前两个视角。因为如果不知道 parameters、tensor 和训练/推理流程是什么，就无法真正理解平台为什么要这样设计。

1.4 张量（tensor）是什么

tensor 可以先简单理解成“有形状的数字数组”。

  • 一个数字是 0 维 tensor。
  • 一列数字是 1 维 tensor。
  • 一张灰度图片可以看成 2 维 tensor。
  • 一批彩色图片可以看成 4 维 tensor，例如 [batch, channel, height, width]。
  • 一批文本 token 也可以看成 tensor，例如 [batch, sequence_length]。

Tensor View

把“文本请求”翻译成“GPU 能计算的张量”

用户看到文本；运行时看到 batch；GPU 最终只看到张量和 kernel。

Input Tensor [32, 2048] 32 条样本，每条 2048 个 token 编号，通常先在 CPU 侧准备。
Embedding
Hidden States [32, 2048, 4096] token 编号变成向量，进入注意力和 MLP 计算。
Kernel
Output Tensor 损失 / 输出概率 训练用损失更新参数；推理用输出概率选择下一个 token。

一针见血地说：AI 框架写的是模型逻辑，GPU 执行的是张量算子，系统工程要保障这些张量能持续、高效、可恢复地流动。

为什么 tensor 重要？因为 GPU 擅长的不是“理解图片”或“理解文本”，而是大规模并行处理 tensor。AI 框架写的是模型逻辑，底层后端执行的是 tensor operator，平台要保障的是 tensor 计算可以持续、高效、可恢复地运行。

1.5 训练是什么

训练的目标是：用数据更新模型参数。

一个训练 step 可以简化为五步：

  1. 读取一批样本。
  2. 执行前向计算，得到预测结果。
  3. 计算损失，衡量预测和真实答案差多远。
  4. 执行反向计算，得到 gradient。
  5. 优化器根据 gradient 更新 parameters。

训练像“反复做题并修改答案”。每一批样本都会让参数移动一点点。训练任务真正消耗的，不只是 GPU 算力，还包括数据读取、显存、通信、checkpoint、日志和失败恢复。

训练时显存里通常会放这些东西：

显存内容 人话解释 为什么占资源
parameters / weights 模型当前学到的数字规则 模型越大，parameters 越多
gradient 告诉 parameters 往哪个方向改的信号 反向计算后需要暂存
activation 每一层计算留下的中间结果 反向计算要用，深模型会很多
优化器状态 优化器保存的额外账本 Adam 等优化器可能保存多份参数级状态

这就是为什么训练任务常常显存压力很大：它不仅要“读模型”，还要“改模型”，并且要保存修改过程中的大量中间信息。

1.6 推理是什么

推理的目标是：用已经训练好的参数生成结果。

推理不再更新 parameters。它更像“拿已经学好的知识快速作答”。用户发来请求，系统把请求转换成 tensor，模型读取 parameters 并计算，最后返回结果。

大语言模型推理还会遇到三个特别重要的问题：

名词 先用人话理解 系统含义
首 token 延迟 用户点击发送后，多久看到第一个字 直接影响聊天、代码补全、流式输出体感
KV Cache 为了记住前文而保存的 key/value 中间结果 上下文越长、并发越高，显存占用越大
batching 把多个请求攒成一个 batch 一起算 能提高 GPU 吞吐，但可能增加等待时间

举一个 batching 的例子。三个用户几乎同时请求模型：

  • 不使用 batching：GPU 分三次处理，简单但浪费并行能力。
  • 固定 batching：系统等凑够一个 batch 再处理，吞吐更高，但第一个用户可能多等一会儿。
  • 动态 batching：系统在很短时间窗口内自动凑请求，尽量兼顾吞吐和延迟。

所以，推理服务看起来像 Web 服务，但核心瓶颈经常不是 HTTP 处理，而是 weights、KV Cache、batching 策略、队列时间和 GPU 计算效率。

1.7 训练和推理的区别

训练和推理都使用模型和 GPU，但它们不是同一种系统。

对比项 训练 推理
核心目标 更新参数 生成结果
输入 大量训练样本 在线请求或批量请求
参数是否变化 会变化 通常不变化
显存重点 parameters、gradient、activation、优化器状态 weights、KV Cache、当前请求中间结果
性能指标 样本吞吐、step time、收敛速度、失败恢复成本 首 token 延迟、吞吐、并发、每 token 成本
常见风险 数据喂不饱 GPU、通信慢、checkpoint 成本高 显存被上下文缓存占满、队列变长、延迟抖动

这个对比非常重要。后面讲训练框架、推理运行时、调度策略和平台架构时，所有差异都会从这里展开。

1.8 为什么会变成系统工程问题

如果只是在一台机器上跑一个很小的模型，训练和推理看起来只是程序问题。可是一旦模型变大、数据变多、用户变多、GPU 变贵，问题就会变成系统工程问题。

AI 任务至少有五条资源流：

资源流 训练中的表现 推理中的表现 关键问题
数据流 样本读取、解码、数据增强 请求输入、分词、模型权重加载 数据能否持续送到计算端
计算流 前向、反向、参数更新 上下文处理、逐 token 生成 GPU 是否在做有效计算
显存流 parameters、gradient、activation、优化器状态 weights、KV Cache、中间结果 显存是否限制 batch size 和并发
通信流 多 GPU gradient 同步、parameters 切分 多副本路由、跨卡模型切分 网络是否拖慢整体
控制流 训练循环、checkpoint、恢复 队列、batching、扩缩容、限流 系统是否能稳定运行

Animated Diagram

一个 batch 经过集群时，真正穿过了哪些瓶颈

蓝点代表一个训练 batch 或一次推理请求的张量负载。

1. 存储 对象存储、本地盘、模型仓库和 checkpoint。慢在这里，GPU 只能等数据。
2. CPU 数据加载、分词、解码、数据增强。CPU 不是主角，但经常卡住入口。
3. GPU kernel 执行前向计算、反向计算、上下文处理和逐 token 生成。这里才是昂贵算力。
4. HBM 参数、中间激活、梯度、优化器状态、上下文缓存。显存决定批大小和并发上限。
5. 网络/控制 GPU 通信、RDMA、服务流量、worker 协调、扩缩容和失败恢复。

GPU 工厂：五条资源流如何一起工作

这不是替代定义的漫画，而是帮助记住资源流的工程图解。

Data Flow

上料线

数据集、对象存储、本地缓存和数据加载进程负责把样本送到计算车间。上料慢，GPU 就会等。

CPU Flow

预处理台

CPU 做解码、分词、数据增强和进程协调。它不是主算力，但经常决定 GPU 是否被喂饱。

计算车间

SM 并行执行工位
HBM 高速显存仓库
Kernel 下发到车间的加工指令

跨厂运输线

GPU 通信库、RDMA、InfiniBand 和 NVLink 承担跨卡、跨节点通信。多机训练慢，常常慢在这里。

Control

运行控制台

训练循环、推理运行时、任务控制器和恢复逻辑决定任务如何推进、何时保存状态、失败后如何继续。

1.9 本章先不讲什么

本章故意不从容器、Pod、CRD、Operator 或调度器开始。原因很简单：这些都是承载层和治理层，不是 AI 系统的起点。

正确的学习顺序应该是：

  1. 先理解 AI 任务本身在做什么。
  2. 再理解训练和推理为什么会消耗不同资源。
  3. 再理解单机任务为什么会扩展成分布式任务。
  4. 最后再看平台和 Kubernetes 如何表达、调度、治理这些任务。

后面的章节会一步步引入 GPU、分布式、平台化和 Kubernetes。到那时再看 Pod、Job、CRD、队列和调度器，它们就不再是孤立名词，而是为了解决前面已经出现的问题。

1.10 常见误区

误区一：AI 系统的核心是模型文件。
模型文件只是 parameters 的持久化形式。真正运行时，还需要数据预处理、tensor 计算、显存管理、请求队列、通信和失败恢复。

误区二：训练和推理只是同一个脚本的两种模式。
不是。训练要更新参数，推理要生成结果。它们的显存构成、性能指标、失败模式和部署形态都不同。

误区三：GPU 利用率高就一定效率高。
不一定。GPU 忙可能是有效计算，也可能是低效底层执行、显存带宽瓶颈、通信等待或排队策略不合理。训练要看 step time 和样本吞吐；推理要看延迟、吞吐、并发和成本。

1.11 本章小结

读完本章，读者应该先建立四个判断：

  • 模型是一组 parameters，tensor 是模型计算的输入和中间形态。
  • 训练的本质是用数据更新参数。
  • 推理的本质是用已有参数生成结果。
  • AI 系统问题会自然扩展成数据、计算、显存、通信和控制五条资源流的问题。

下一章开始，我们先把 tensor 计算讲清楚。只有理解 tensor 和 GPU 为什么匹配，后面讨论训练效率、推理速度、分布式通信和平台调度才有根。
