2  张量计算的本质

2.1 本章要回答的问题

上一章说过，AI 系统会把数据变成张量，再把张量送进模型计算。本章继续追问：张量到底是什么，为什么 AI 计算最后会变成 GPU 上的大量张量计算？

这一章不先讲框架，也不先讲部署。我们先把最基础的计算直觉建立起来：

tensor 是有形状的数字数据；模型计算就是让这些数字按照固定规则反复做乘法、加法、归一化、筛选和重排。

理解 tensor，才能理解后面的 batch、显存、GPU、分布式通信和性能优化。至于 kernel、warp、SM 这些 GPU 执行细节，会在第四篇正式展开。本章先把 kernel 降级成一个够用的理解：它是 GPU 端被启动执行的一段底层计算程序；现在不需要记住 grid、block、thread 的细节，只要知道高层 operator 最终会落到硬件可执行的实现上。

2.2 张量（tensor）是什么

tensor 可以先理解成“有形状的数字数组”。它不是神秘概念，只是比普通数组更强调 shape、dtype、device 和自动求导这些工程属性。PyTorch 教程把 tensor 描述为类似数组和矩阵的专门数据结构，用于编码模型的 input、output 和 parameters，并且可以运行在 GPU 等加速器上 (《Tensors》, 不详)。

Shape Map

张量不是抽象名词，而是“有形状的数据”

形状决定一次计算有多少数据、多少并行度、多少显存占用。

0 维：标量 一个数字，例如 loss=0.37。
1 维：向量 一列数字，例如一个 token 的向量表示。
2 维：矩阵 一张数字表，例如线性层的权重。
更高维：批量数据 一批图片或文本，例如 [batch, seq, hidden]。

几个常见例子：

数据 tensor shape 示例 含义
一张灰度图片 [height, width] 每个位置是一个像素值
一批彩色图片 [batch, channel, height, width] 一批图片，每张图片有多个颜色通道
一批文本 token [batch, sequence_length] 一批文本，每条文本有一串 token 编号
token 向量表示 [batch, sequence_length, hidden_size] 每个 token 被表示成一个向量
线性层权重 [input_size, output_size] 模型参数矩阵

这里的 batch 是“一批样本”的意思。一次只处理一个样本，GPU 并行能力很难发挥；一次处理一个 batch，GPU 可以把大量相似计算一起做。

2.3 形状为什么重要

tensor shape 不是算法细枝末节。对系统工程来说，shape 会直接决定四件事：

  1. 有多少计算量。 形状越大，参与计算的数字越多。
  2. 占多少显存。 每个数字都要占空间，形状越大，中间结果越大。
  3. 能否并行。 形状合适时，GPU 可以把很多计算同时铺开。
  4. 数据怎么搬。 不同布局会影响显存读取是否连续，进而影响速度。

举个例子，一个文本 tensor 的 shape 是 [32, 2048, 4096]：

  • 32 表示一批里有 32 条样本。
  • 2048 表示每条样本有 2048 个 token 位置。
  • 4096 表示每个 token 用 4096 维向量表示。

这个 tensor 一共有 32 x 2048 x 4096 个数字。即使每个数字只占 2 字节，也会占用大量显存。后面训练和推理的很多优化，本质上都在管理这些数字的计算和存放。

2.4 为什么矩阵乘法是核心

神经网络里有很多操作，但矩阵乘法是最核心的一类。原因很简单：模型 parameters 通常就是矩阵，输入数据也会变成矩阵或高维 tensor。模型计算经常是在做“输入乘以权重”。

线性层可以写成：

Y = XW + b

这里：

  • X 是输入 tensor。
  • W 是模型权重。
  • b 是偏置。
  • Y 是输出。

Animated Computation

矩阵乘法：一行乘一列，得到一个输出格子

被高亮的一行和一列，会共同决定右侧绿色输出单元。

a11 a12 a13 a21 a22 a23
x
b11 b12 b21 b22 b31 b32
=
c11 c12 c21 c22

直觉上，矩阵乘法就是把输入数据和模型权重反复做“乘加”。神经网络之所以适合 GPU，是因为这样的乘加可以大规模并行执行。

大语言模型里的注意力计算、前馈网络、投影层，也大量依赖矩阵乘法。实际系统里会有 FlashAttention、算子融合、低精度计算等优化，但底层问题没有变：大量数字要在有限显存和 GPU 计算单元之间高效流动。

2.5 一个 batch 如何变成计算

把 batch 理解成“多条样本合在一起算”，会更容易理解训练和推理。

假设一次处理 32 条文本，每条文本 2048 个 token，每个 token 用 4096 维向量表示。模型并不是一条一条慢慢算，而是把它们组织成 tensor，让 GPU 同时处理大量位置上的计算。

Tensor View

把“文本请求”翻译成“GPU 能计算的张量”

用户看到文本；运行时看到 batch；GPU 最终只看到张量和 kernel。

Input Tensor [32, 2048] 32 条样本，每条 2048 个 token 编号，通常先在 CPU 侧准备。
Embedding
Hidden States [32, 2048, 4096] token 编号变成向量，进入注意力和 MLP 计算。
Kernel
Output Tensor 损失 / 输出概率 训练用损失更新参数；推理用输出概率选择下一个 token。

一针见血地说：AI 框架写的是模型逻辑，GPU 执行的是张量算子，系统工程要保障这些张量能持续、高效、可恢复地流动。

这就是 batch 带来的好处：它把很多相似的小计算组织成大块计算，让 GPU 更容易发挥并行能力。

但 batch 不是越大越好。它会带来三种代价：

代价 表现 后果
显存占用变大 中间结果、缓存和输出都变大 batch 太大可能直接 OOM
数据供给压力变大 CPU、存储、解码和预处理要更快 数据慢会让 GPU 空等
延迟可能变大 推理请求要等待凑批 用户可能更晚看到结果

所以 batch 既是算法参数，也是系统参数。训练关注它对吞吐和收敛的影响，推理关注它对吞吐和延迟的平衡。

2.6 计算不是唯一成本

很多人第一次理解 AI 计算时，会以为瓶颈一定在“算得不够快”。实际并不总是这样。

一次 tensor 计算至少包含三类成本：

成本 含义 典型瓶颈
计算成本 执行乘法、加法、归一化、激活等操作 GPU 算力、Tensor Core、operator 实现
内存成本 从显存读取输入、权重和中间结果，再写回输出 HBM 带宽、缓存命中、数据布局
调度成本 启动和安排底层执行任务、处理依赖 后端调度、CPU-GPU 同步、过多小算子

有些操作计算很多、数据复用高，叫计算密集型。
有些操作计算不多，但频繁读写大量数据，叫内存带宽受限。

这就是为什么只看 GPU 利用率不够。GPU 忙不忙是一回事，忙得是否有效是另一回事。

2.7 从模型代码到实际执行

在 PyTorch 或 TensorFlow 里，用户写的是高层模型代码，例如：

Y = XW + b

但机器不会直接理解“线性层”这个概念。框架会把高层代码拆成一组 operator，也就是 tensor 操作，例如矩阵乘法、加法、归一化、激活函数、softmax、attention 等。

到这里先不要急着理解 kernel。更适合本章的说法是：

用户写的是模型逻辑；框架看到的是 operator；底层后端执行的是这些 operator 对应的高性能实现。

如果 tensor 在 CPU 上，后端会调用 CPU 侧实现；如果 tensor 在 NVIDIA GPU 上，后端通常会经过 CUDA、cuBLAS、cuDNN 等库，把计算交给 GPU 执行。CUDA 编程指南把在 GPU 设备上执行的函数称为 kernel，并说明 kernel 会按 grid、block、thread 的层次启动 (《CUDA C++ Programming Guide》, 不详)。这些 GPU 结构会在下一章讲清楚。

2.8 训练和推理对显存的要求不同

训练和推理都使用 tensor，但它们保留的 tensor 不一样。

训练要更新参数，因此需要保存更多中间信息：

  • 模型参数。
  • 前向计算产生的 activation，也就是中间激活。
  • 反向计算得到的 gradient，也就是梯度。
  • 优化器状态，例如 Adam 的一阶、二阶动量。
  • 临时 buffer、通信 buffer 和运行时开销。

推理通常不更新参数，所以没有梯度和优化器状态。但大语言模型推理会引入另一类显存压力：上下文缓存，也就是 KV Cache。上下文越长、并发越高，上下文缓存越可能成为服务容量上限。

这也是为什么训练优化和推理优化不能套用同一套指标。训练常看样本吞吐和 step time；推理常看首 token 延迟、输出速度、并发和每 token 成本。

2.9 常见误区

误区一：模型写在 Python 里，所以 Python 一定是主要性能瓶颈。
大多数核心计算会交给底层后端实现。Python 可能影响数据处理和调度，但矩阵乘法、卷积、attention 等重计算通常由高性能库和硬件加速执行。

误区二：矩阵乘法快，模型就一定快。
不一定。真实模型还会受显存带宽、数据布局、operator 调度、同步、数据加载和通信影响。

误区三：batch 越大越好。
batch 变大会增加显存占用，也可能放大通信压力；推理里还会增加等待时间。

误区四：推理显存只由模型权重决定。
对大语言模型推理来说，上下文缓存会随上下文长度和并发数增长，常常决定服务容量上限。

2.10 本章小结

读完本章，需要记住五个判断：

  • tensor 是有形状的数据，不是抽象黑话。
  • tensor shape 决定计算量、显存占用、并行度和数据搬运方式。
  • 矩阵乘法重要，是因为模型参数和输入数据经常以矩阵形式参与计算。
  • AI 性能不只取决于算力，还取决于显存访问、operator 调度、底层执行和数据供给。
  • 训练和推理保留的 tensor 不同，所以显存压力和优化目标也不同。

下一章会继续讲模型、parameters 与显存：tensor 计算为什么会消耗显存，训练和推理分别把显存花在什么地方。GPU 的执行结构会在第四篇系统展开。
