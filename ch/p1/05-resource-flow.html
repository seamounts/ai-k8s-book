5  AI 系统的五条资源流

5.1 本章要回答的问题

前面已经知道，AI 系统不是“一个脚本加一张 GPU”。现在要把它整理成平台工程能分析的结构。

本章给出一个贯穿全书的模型：

AI workload 是数据流、计算流、显存流、通信流和控制流的组合。任何一条流断了，GPU 都可能空等；任何一条流设计错了，训练效率或推理体验都会被拖垮。

5.2 五条资源流

资源流 训练中的表现 推理中的表现 典型问题
数据流 数据集读取、shuffle、解码、增强 请求输入、tokenizer、模型权重加载 数据是否持续送到计算端
计算流 forward、loss、backward、optimizer prefill、decode、后处理 GPU 是否在做有效计算
显存流 weights、activation、gradient、优化器状态 weights、KV Cache、请求状态 显存是否限制 batch 或并发
通信流 gradient 同步、参数切分、checkpoint 写入 请求路由、跨 GPU 模型切分 网络是否拖慢整体
控制流 训练循环、失败恢复、实验记录 队列、batching、限流、扩缩容 系统是否能稳定推进

Animated Diagram

一个 batch 经过集群时，真正穿过了哪些瓶颈

蓝点代表一个训练 batch 或一次推理请求的张量负载。

1. 存储 对象存储、本地盘、模型仓库和 checkpoint。慢在这里，GPU 只能等数据。
2. CPU 数据加载、分词、解码、数据增强。CPU 不是主角，但经常卡住入口。
3. GPU kernel 执行前向计算、反向计算、上下文处理和逐 token 生成。这里才是昂贵算力。
4. HBM 参数、中间激活、梯度、优化器状态、上下文缓存。显存决定批大小和并发上限。
5. 网络/控制 GPU 通信、RDMA、服务流量、worker 协调、扩缩容和失败恢复。

动态变化

入口变慢：GPU 空等 显存顶满：batch 或并发上不去 网络拥塞：多卡互相等待 控制流缺失：失败恢复成本高

5.3 为什么这比“看 GPU 利用率”更可靠

GPU 利用率只是计算流的一个侧面。一个任务 GPU 利用率低，可能是数据流慢；一个任务 GPU 利用率高，可能是低效 operator 在忙；一个服务吞吐高，也可能是 batching 过度导致首 token 延迟变差。

Performance Reading

GPU 忙，不等于系统快

判断瓶颈时，要把计算、显存、数据和通信放在同一张图里看。

Compute-bound 算力受限

矩阵乘法、卷积、注意力等计算量很大。优化重点通常是低精度、Tensor Core、算子融合和更合适的 shape。

Memory-bound 带宽受限

读写数据比计算更贵。优化重点是减少搬运、提高数据复用、改善布局、减少不必要的中间张量。

Input-bound 供给受限

CPU、数据加载、分词、解码、远端存储跟不上。GPU 指标看起来低，根因却不在 GPU 本身。

Network-bound 通信受限

多卡训练等 AllReduce，多节点推理等模型分片通信。优化重点是拓扑、RDMA、并行策略和通信重叠。

所以，平台排障不能只问“GPU 忙不忙”，而要问：

  • 数据是否及时到达？
  • operator 是否足够成块、足够高效？
  • 显存账本是否合理？
  • 多 GPU 或多节点通信是否拖慢整体？
  • 控制系统是否能根据状态做出正确动作？

这五个问题会贯穿后面的训练流程、推理流程、GPU 性能、分布式 AI、Kubernetes 控制器和调度系统。

5.4 五条资源流如何互相影响

五条资源流不是并列清单，而是互相牵引的系统。

触发点 表面现象 背后的连锁反应
数据流慢 GPU 利用率低 计算流空等，训练 step time 变长，队列资源被占住
显存不足 batch size 上不去 计算流不能成块，吞吐下降，推理并发被限制
通信慢 多卡扩展效率差 GPU 等 AllReduce 或跨卡同步，算力无法线性放大
控制流弱 失败恢复慢 checkpoint、重试、抢占和扩缩容都变得危险
计算流低效 GPU 很忙但吞吐低 operator 小而碎、访存差或同步多，成本被浪费

这也是为什么平台优化很少是单点优化。比如把 GPU 从 A10 换成 H100，如果数据加载、通信和显存账本没有一起优化，整体速度未必按硬件峰值提升。反过来，如果 DataLoader、缓存、batch shape 和通信重叠做好了，同样的 GPU 可能跑出完全不同的效率。

5.5 一个排障案例：GPU 只有 20%

假设一个训练任务申请了 8 张 GPU，但监控显示 GPU 利用率长期只有 20%。这时不要马上得出“调度器分配错了”或“GPU 太差”的结论。按照五条资源流，可以更快定位问题。

观察 可能对应的资源流 下一步验证
GPU 计算一阵、停一阵 数据流 看 DataLoader 等待时间、CPU 使用率、远端存储吞吐
每个 step 都卡在 backward 后面 通信流 看 AllReduce 时间、网卡吞吐、GPU 拓扑和跨节点比例
batch size 一加就 OOM 显存流 看 activation、optimizer state、临时 buffer 和显存碎片
GPU 利用率高但 tokens/sec 不高 计算流或显存流 看 operator 时间、HBM 带宽、shape 是否导致小算子过多
checkpoint 时整体停顿 控制流和数据流 看保存频率、对象存储延迟、异步 checkpoint 能否使用

这个案例说明一件事：GPU 利用率是结果，不是诊断框架。真正的诊断框架是五条资源流。后面讲 Kubernetes 调度时，我们才会把这些信号变成队列、优先级、节点选择、拓扑约束和弹性策略。

5.6 平台观测要如何对应五条流

后面进入生产平台时，可以把监控指标先映射到五条流：

资源流 应该观测什么 典型工具或信号
数据流 读取吞吐、DataLoader 等待、tokenizer 时间、缓存命中率 训练日志、数据管道指标、对象存储指标
计算流 GPU 利用率、kernel 时间、operator 分布、tokens/sec DCGM、Profiler、runtime metrics
显存流 allocated/reserved、KV Cache、OOM、碎片、batch size GPU metrics、runtime cache metrics
通信流 AllReduce 时间、NCCL 错误、RDMA/NIC 吞吐、跨节点比例 NCCL logs、网络指标、拓扑信息
控制流 queue time、重启次数、checkpoint 时间、扩缩容延迟 调度器、Operator、服务网关、事件日志

这些指标不是为了堆仪表盘，而是为了让排障能回到因果链：哪个资源流先出问题，它如何拖慢其他流。

5.7 本章小结

  • 数据流决定 GPU 是否被喂饱。
  • 计算流决定 tensor 是否被高效处理。
  • 显存流决定 batch size、上下文长度和并发上限。
  • 通信流决定分布式训练和多 GPU 推理的扩展效率。
  • 控制流决定任务能否长期稳定运行、失败后能否恢复。
