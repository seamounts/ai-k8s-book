4  训练与推理的根本差异

4.1 本章要回答的问题

训练和推理经常共享同一个模型文件、同一类 GPU、同一套容器镜像，甚至同一套基础设施。但它们不是同一种系统。

训练的核心目标是让模型变好。推理的核心目标是让模型可用。前者关注参数更新、吞吐、收敛和失败恢复；后者关注请求延迟、并发、稳定性和单位 token 成本。

本章给出全书第二条判断标准：

训练是以数据驱动参数更新的离线计算系统；推理是以请求驱动结果生成的在线服务系统。

这句话会贯穿后面的框架选择、部署架构、调度策略和性能优化。

4.2 一张图看清两种系统形态

Training Step

一个训练 step 如何推进

训练不是一次 forward，而是数据、计算、状态和持久化共同推进。

Data 读取样本

Dataset/DataLoader 从存储中读取样本，执行 shuffle、transform、tokenizer。

Batch 形成 batch

collate 把多条样本组织成 tensor，padding 和 shape 会影响显存。

Forward 前向计算

模型用当前 parameters 计算 output，并留下 activation。

Loss 衡量误差

loss 把 output 与 target 的差距变成可优化目标。

Backward 产生 gradient

autograd 根据计算图反向传播，计算每个 parameter 的 gradient。

Update 更新状态

optimizer 更新 parameters；checkpoint 定期保存可恢复状态。

平台排障时，不要只看 GPU。DataLoader、CPU、显存、通信、checkpoint 写入和指标记录都会影响 step time。

Inference Path

一次 LLM 请求如何变成流式输出

推理服务的核心状态，是队列、weights、KV Cache 和正在生成的请求。

Request 用户请求

请求进入 router 或 gateway，携带 prompt、参数和超时约束。

Queue 排队与 batching

系统在有限等待时间内合并请求，提高 GPU 吞吐。

Tokenizer 文本转 token

prompt 被转换成 token id，这一步通常消耗 CPU。

Prefill 处理上下文

GPU 处理完整 prompt，建立初始 KV Cache，决定 TTFT 的大头。

Decode 逐 token 生成

每一步读取 KV Cache 并生成下一个 token，影响 TPOT。

Stream 流式返回

生成的 token 持续返回给用户，请求结束后释放相关状态。

LLM 推理不是无状态 Web 请求。上下文越长、并发越高，KV Cache 越可能成为显存瓶颈。

训练像一条长期运行的生产线。它把大量数据一批一批送入模型，不断计算梯度并更新参数。一次训练可能运行数小时、数天甚至更久。

推理像一个在线服务。它等待请求到来，加载已经训练好的参数，尽快产生结果。一次请求可能只持续几十毫秒，也可能在大模型场景下持续生成数十秒。

4.3 训练：参数更新系统

训练的主循环是：

data -> forward -> loss -> backward -> optimizer -> updated parameters

这个循环决定了训练系统的几个特征。

第一，训练需要保存中间状态。backward 依赖 forward 产生的 activation，所以训练显存压力远高于同等规模的推理。

第二，训练需要容错。长时间训练如果中途失败，不能简单从头开始。checkpoint、resume、实验追踪和数据版本管理，都是训练系统的一部分。

第三，分布式训练需要协同。多个 worker 之间通常要同步梯度、切分参数或传递中间结果。任何一个 worker 慢下来，都可能拖慢整体。

第四，训练的性能指标更偏向吞吐和有效进展。常见指标包括 step time、samples/sec、tokens/sec、GPU 利用率、通信时间、数据加载时间和收敛曲线。

4.4 推理：结果生成系统

推理的主路径是：

request -> queue/router -> model runtime -> output

对传统模型，推理通常是输入一次、执行一次、返回一次。对大语言模型，推理还要拆成 prefill 和 decode：

阶段 人话解释 系统影响
prefill 先把用户输入的完整 prompt 处理一遍 建立初始 KV Cache，通常决定首 token 延迟的大头
decode 每次生成一个新 token，并把它接回上下文 影响每个后续 token 的速度、显存占用和流式体验

推理系统的特征也很明确。

第一，推理关注用户体感。首 token 延迟、P95/P99、流式输出稳定性，比单纯 GPU 利用率更重要。

第二，推理关注并发。模型权重常驻显存，请求共享同一批模型副本。系统要在延迟和吞吐之间找到平衡。

第三，LLM 推理显存压力来自权重和 KV Cache。并发越高、上下文越长，KV Cache 越容易成为容量上限。

第四，推理需要在线治理。灰度、回滚、多模型路由、扩缩容、限流、熔断和观测，都是推理服务的生产问题。

4.5 指标不同，优化目标不同

训练和推理最容易混淆的地方，是都能看到 GPU 利用率。但同一个 GPU 利用率，在两类系统中含义不同。

维度 训练 推理
输入来源 数据集、样本流 用户请求、服务调用
核心目标 更新参数、提升模型质量 生成结果、满足 SLA
主要瓶颈 显存、数据加载、通信、计算 延迟、并发、KV Cache、batching
关键指标 step time、samples/sec、收敛速度 TTFT、TPOT、tokens/sec、P95/P99
失败处理 checkpoint resume、job restart 副本摘除、请求重试、回滚
调度重点 Gang Scheduling、队列、公平性、拓扑 低延迟、副本分布、弹性伸缩
成本视角 单位训练进展成本 单位请求或 token 成本

这里的 batching，是指把多个样本或多个请求合并成一批一起计算。训练里的 batch 主要服务于吞吐和收敛；推理里的 batching 主要是在吞吐和用户等待之间做取舍。

不要把“batching”理解成一个抽象优化词。它本质上是在问：GPU 一次启动计算时，要不要把多份相似工作放在同一块 tensor 里一起算？NVIDIA Triton Inference Server 的动态 batching 文档就是这个思路：服务器把多个独立推理请求动态合并成一个 batch，并通过最大排队等待时间控制延迟 (《Triton Inference Server: Dynamic Batcher》, 不详)。

同一个词，在训练和推理里重点不同：

场景 batching 解决什么 最大风险
训练 把更多样本合成大块 tensor，提高 GPU 吞吐和统计稳定性 batch 太大导致 OOM，或改变收敛行为
推理 把多个在线请求合并计算，提高 GPU 吞吐和单位成本 请求等待变长，TTFT 和 P99 变差

训练优化经常愿意牺牲部分单步延迟换取整体吞吐。推理优化不能只追求吞吐，因为用户会直接感受到排队和首 token 延迟。

4.6 为什么不能用一种调度策略覆盖两者

训练任务通常是批处理任务。它可以排队，可以等待资源齐备，可以在有 checkpoint 的情况下被抢占，也可以通过 backfilling 利用空档。

推理服务通常是在线服务。它需要稳定副本，需要快速扩容，需要避免冷启动冲击，需要把请求持续路由到健康实例。

这导致两类任务对调度系统提出不同要求：

  • 训练更关心资源成组到位、队列公平性和长任务治理。
  • 推理更关心副本可用性、延迟稳定性和弹性伸缩。
  • 分布式训练对 GPU 拓扑敏感，普通小模型推理可能更适合分散多副本。
  • 大模型推理在多卡或多节点部署时，也会重新引入拓扑和 gang 类问题。

所以，生产集群里常见三种策略：

  1. 训练集群和推理集群物理隔离。
  2. 训练和推理共享集群，但使用不同队列、节点池、优先级和抢占规则。
  3. 对在线推理保留稳定资源，对离线训练使用剩余资源和可抢占资源。

没有绝对正确的答案，只有和业务目标匹配的资源治理方式。

4.7 从流程差异到系统设计

现在还不需要展开具体平台对象。先抓住更底层的差异：训练和推理不是“运行方式不同”，而是生命周期、状态、失败处理和指标目标都不同。

这个差异会自然推出后面的系统设计：

  • 训练更像有终点的长作业：需要启动一组 worker，持续推进 step，定期保存 checkpoint，失败后尽量从最近状态恢复。
  • 推理更像无终点的在线服务：需要稳定接收请求，控制延迟，处理扩缩容、灰度、回滚和健康检查。
  • 训练的资源诉求常常是“同时给齐一组 GPU”，否则分布式任务无法有效开始。
  • 推理的资源诉求常常是“让足够副本持续可用”，否则用户请求会排队、超时或失败。

后面进入 Kubernetes 时，会把这些抽象差异映射到 Job、CRD、Operator、Deployment、Service、Gateway、队列、Gang Scheduling 和弹性伸缩上。但在本章，最重要的是先看懂工作负载本身，而不是急着看 YAML。

4.8 常见误区

误区一：训练脚本能跑，就能变成生产训练任务。
生产训练还需要数据版本、checkpoint、分布式启动、失败恢复、指标和调度策略。

误区二：推理服务能返回结果，就算部署完成。
生产推理还要看延迟、并发、扩缩容、灰度、回滚、观测和成本。

误区三：训练和推理都用 GPU，所以可以使用同一套利用率指标。
GPU 指标只是底层信号。训练要结合 step time 和收敛；推理要结合 TTFT、TPOT、P95/P99 和 tokens/sec。

误区四：推理总比训练简单。
小模型推理可能简单，大模型推理在 KV Cache、batching、流式输出和多副本调度上非常复杂。

4.9 本章小结

读完本章，需要形成三个判断：

  • 训练是参数更新系统，推理是结果生成系统。
  • 训练关注长期吞吐和有效进展，推理关注在线延迟和并发稳定性。
  • 两者即使共享基础设施，部署、调度和观测语义也完全不同。

后面讨论训练框架、推理运行时和 Kubernetes CRD 时，都要先判断它服务的是哪一种系统形态。
