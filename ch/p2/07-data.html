7  数据准备与样本组织

7.1 本章要回答的问题

训练流程的第一步不是写模型，而是准备数据。模型不会直接从“图片文件”“文本文件”“日志记录”里学习，它只能从被组织好的样本和 label 中学习。

数据准备的本质，是把真实世界的原始记录整理成可重复读取、可追踪版本、可转换成 tensor 的训练样本。

7.2 样本是什么

一个训练样本通常包含三部分：

部分 图像分类例子 语言模型例子 系统含义
input 图片像素 token 序列 会被转换成 tensor
target / label 类别编号 下一个 token 或监督答案 用于计算 loss
metadata 文件路径、来源、版本 文档 ID、语言、时间 用于过滤、审计和复现

数据准备最怕“能跑但不可复现”。如果数据版本、清洗规则、采样逻辑和 label 来源不可追踪，训练结果就很难解释，后续回滚、比较和排障也会失去依据。

Excalidraw Style

训练数据契约：原始记录如何变成 sample

手绘风格源文件：fig/excalidraw/training-data-contract.excalidraw

Raw 原始记录 图片、文本、日志、表格行。此时还不是训练样本，只是业务世界留下的数据。
Manifest 样本清单 记录路径、label、版本、过滤规则和来源，让训练结果可以回溯。
Sample 样本契约 每条样本必须明确 input、target 和 metadata，训练循环才知道如何计算 loss。
Batch 批量 tensor 多条 sample 被 collate 成 shape 明确的 tensor，然后送入模型。
input模型要看的内容，例如图片像素、token id、特征向量。
targetloss 要对齐的答案，例如类别、下一个 token、偏好标签。
metadata复现和治理所需的信息，例如版本、来源、时间、语言、权限。

这里要抓住一个词：契约。训练数据不是“有一堆文件就行”，而是每条 sample 必须清楚回答三个问题：

  • 模型看什么，也就是 input。
  • loss 对什么，也就是 target。
  • 结果如何追溯，也就是 metadata。

如果这三个问题没有固定下来，后面的 DataLoader、loss、指标、checkpoint 和模型发布都会变得含糊。比如同样是图片分类，目录名可以当 label，CSV 也可以当 label；但如果训练时混用了两套 label 来源，loss 曲线仍然可能下降，模型质量却已经不可解释。

7.3 数据集组织方式

常见组织方式包括：

  • 文件目录：适合图片、小规模实验，直观但元数据能力弱。
  • 表格/manifest：用 CSV、Parquet、JSONL 记录样本路径和 label，更适合大规模训练。
  • 数据湖/对象存储：适合海量样本，但要处理吞吐、缓存和权限。
  • 流式数据：适合持续训练或在线学习，但版本和复现更复杂。

平台工程要关心的不是“数据放在哪里”这么简单，而是：

  • 多个 worker 是否能并发读取？
  • 数据顺序是否可控？
  • 随机性是否可复现？
  • 样本是否能被分片？
  • 数据版本和训练产物是否能关联？

7.4 一个具体例子：图片分类 manifest

一个可治理的数据集，通常会有一份显式 manifest，而不是只依赖目录结构：

sample_id,path,label,split,version,source
img-0001,s3://bucket/cat/0001.jpg,cat,train,2026-06-01,camera-a
img-0002,s3://bucket/dog/0002.jpg,dog,train,2026-06-01,camera-b
img-9101,s3://bucket/cat/9101.jpg,cat,eval,2026-06-01,camera-a

这份 manifest 的价值不是“多写一张表”，而是让训练从一开始就具备三种能力：

能力 为什么重要
可复现 同一次实验可以重新读到同一批样本和同一套 label
可比较 两个模型可以确认是否使用了同一数据版本
可审计 发现坏样本、脏 label 或数据泄漏时，可以追溯来源

对平台来说，数据版本应该和 checkpoint、模型产物、实验记录关联在一起。否则模型上线后出问题，很难判断是代码变了、参数变了，还是数据变了。

7.5 常见数据问题

问题 表现 后果
数据泄漏 eval 样本出现在 train 里 指标虚高，上线效果差
label 漂移 同一类样本在不同版本里标注规则变了 实验之间不可比较
随机性不可控 shuffle、采样、增强没有记录 seed 训练结果难以复现
worker 重复读取 分布式 worker 没有正确分片 有效数据量变少，收敛判断失真
小文件过多 远端存储元数据请求爆炸 GPU 等数据，step time 抖动

7.6 数据切分不是随机分一下

训练数据通常会切成 train、validation、test 三部分。最常见的错误，是只做随机切分，却没有考虑数据之间的相关性。

场景 错误切分 更合理的切分
用户行为模型 同一个用户的数据同时出现在 train 和 eval 按用户或时间切分
文档问答 同一篇文档的不同片段分到 train 和 eval 按文档 ID 切分
图像分类 同一视频抽帧进入不同 split 按视频或采集批次切分
日志异常检测 未来时间的数据进入训练集 严格按时间切分

切分策略会直接影响指标可信度。平台不能只保存“数据路径”，还要保存 split 规则、过滤规则、采样规则和随机种子。否则 eval score 可能很好看，但只是因为评测集和训练集泄漏了相同来源。

7.7 数据质量要前置，不要等训练后才发现

训练前至少应该做四类检查：

检查 发现什么问题
schema 检查 字段缺失、类型错误、label 非法
分布检查 类别极不均衡、长度长尾、异常值
重复检查 样本重复、近重复、跨 split 泄漏
合规检查 数据来源、授权、隐私字段、脱敏状态

这些检查听起来像数据工程问题，但它们会直接变成训练问题：脏 label 会让 loss 学错方向，重复样本会让模型过拟合，长度长尾会让 batch padding 浪费显存，数据许可不清会让模型产物无法发布。
