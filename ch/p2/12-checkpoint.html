12  Checkpoint、恢复与容错

12.1 本章要回答的问题

长训练任务一定要考虑失败。节点故障、Pod 重启、网络中断、OOM、抢占都会发生。没有 checkpoint，失败就意味着大量 GPU 时间作废。

PyTorch 保存和加载模型教程强调，通用 checkpoint 往往不只保存模型 state_dict，还会保存 optimizer state、epoch、loss 等训练状态 (《Saving and Loading Models》, 不详)。

Excalidraw Style

checkpoint 是长训练任务的存档点

手绘风格源文件：fig/excalidraw/checkpoint-resume.excalidraw

Step 1000 正常训练 持续读取数据、计算 gradient、更新 parameters。
Save 保存 checkpoint model、optimizer、scheduler、step、random state、data cursor。
Failure 节点/进程失败 OOM、抢占、网络故障、机器重启都可能发生。
Resume 从最近状态继续 损失的是 checkpoint 间隔内的进展，而不是整个训练任务。
间隔太长失败损失大，抢占成本高。
间隔太短存储写入频繁，step time 抖动。
写入太慢需要异步保存、分片保存或更近的存储层。

12.2 checkpoint 应该保存什么

内容 为什么需要
model state 保存 parameters
optimizer state 恢复更新轨迹
scheduler state 恢复 learning rate
step / epoch 恢复训练进度
random state 尽量保证复现
数据状态 避免重复或跳过样本

只保存 weights，通常只能用于推理或重新开始训练；要无缝 resume，必须保存更完整的训练状态。

12.3 checkpoint 的系统代价

checkpoint 不是免费操作：

  • 写入大文件会占用存储带宽。
  • 多 worker 同时写可能冲击文件系统。
  • 频率太低，失败损失大。
  • 频率太高，训练 step 抖动。
  • 远端对象存储可能带来长尾延迟。

平台需要在恢复成本和运行开销之间取平衡。

一个简单判断是：

期望损失 = 失败概率 x checkpoint 间隔内的平均训练成本
运行开销 = checkpoint 写入时间 + 存储带宽 + 对训练 step 的扰动

checkpoint 策略不是越频繁越好，也不是越少越好。它要跟任务时长、GPU 成本、存储能力、抢占策略和业务恢复目标一起设计。

12.4 checkpoint 与调度的关系

checkpoint 不只是训练框架功能，也会影响调度策略：

  • 没有 checkpoint 的长训练任务，不适合被频繁抢占。
  • checkpoint 间隔越长，被抢占或失败时损失越大。
  • checkpoint 写入越慢，训练 step 抖动越明显。
  • 多 worker checkpoint 如果没有协调，可能产生不一致状态。

后面讲 Kueue、Volcano、抢占和机会型任务时，会反复回到这一点：可恢复性决定任务能否被安全调度。

12.5 单文件 checkpoint 与 sharded checkpoint

小模型训练常见的是单文件 checkpoint。大模型训练更常见的是 sharded checkpoint，也就是每个 rank 保存自己负责的一部分状态。

类型 优点 风险
单文件 checkpoint 简单，易于复制和加载 大模型下写入慢，单点文件巨大
sharded checkpoint 并行写入，适合参数和 optimizer state 切分 文件数量多，恢复依赖并行配置和元数据

sharded checkpoint 对平台提出更高要求：元数据要完整，写入要么成功要么可识别失败，清理策略不能误删部分分片，恢复时要能找到同一版本的所有分片。

12.6 checkpoint 写入的一致性

一次 checkpoint 至少要保证三件事：

  1. 读到的是同一个训练 step 的状态。
  2. 所有 worker 的分片属于同一次保存。
  3. 失败的半成品不会被当作可恢复 checkpoint。

实践中通常会使用临时目录、完成标记、manifest 和原子切换来表达“这个 checkpoint 已完成”。否则故障恢复时可能加载到半写入状态，问题会非常隐蔽。

12.7 本章小结

checkpoint 是训练任务从“能跑”走向“可恢复”的关键。后面讲抢占、队列和机会型训练时，checkpoint 是前提能力。
