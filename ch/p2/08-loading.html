8  数据加载与预处理

8.1 本章要回答的问题

数据准备好以后，还不能直接进入模型。训练进程需要把样本读取出来，执行解码、清洗、增强、tokenizer、padding、collate，最后形成一个 batch。

PyTorch 官方教程把 Dataset 和 DataLoader 分开：Dataset 负责保存样本和 label，DataLoader 负责围绕 Dataset 提供可迭代访问，并支持 batching、shuffle、多进程加载等能力 (《Datasets and DataLoaders》, 不详)。这正好说明一个关键分层：数据是什么和数据如何高效送进训练循环是两个问题。

8.2 数据加载链路

storage -> Dataset -> transform/tokenizer -> collate -> batch tensor -> model

Excalidraw Style

DataLoader 像一条给 GPU 上料的传送带

手绘风格源文件：fig/excalidraw/dataloader-path.excalidraw

Storage 远端/本地数据 对象存储、文件系统、Parquet、JSONL、小文件目录。
Dataset 定义样本访问 map-style 按 index 取；iterable-style 像流一样产生。
Workers 并发读取 多进程加载、预取、缓存，减少训练进程等待。
Transform 解码/增强/分词 CPU 常见热点：图像解码、tokenizer、数据增强。
Collate 拼 batch padding、对齐 shape、合并 tensor，决定进入模型的形状。
GPU 开始训练 step 数据到得稳定，GPU 才能持续做有效计算。

排障口诀：GPU 空等时，先看这条传送带有没有断；不要一上来只调 batch size 或怀疑调度器。

每一段都可能成为瓶颈：

阶段 做什么 常见瓶颈
storage 读取文件、对象、表格 远端存储吞吐不足、元数据慢
transform 解码、裁剪、增强、清洗 CPU 不足、单线程处理
tokenizer 文本转 token tokenizer CPU 开销大
collate 拼成 batch、padding shape 不规则、内存拷贝多
transfer CPU 到 GPU 数据移动 pinned memory、同步等待

GPU 空等，很多时候不是 GPU 本身慢，而是数据没有及时到达。

这条链路要和第一篇的五条资源流连起来看：storage、Dataset、transform 属于数据流；collate 会改变 tensor shape，影响显存流；transfer 会涉及 CPU 内存、pinned memory 和 GPU 显存；如果分布式 worker 读取远端存储，还会把通信流和数据流绑在一起。

8.3 Map-style 与 Iterable-style 数据

训练数据并不总是“第 i 个样本可以随机读取”。PyTorch 文档把数据集分成 map-style 和 iterable-style 两类 (《Datasets and DataLoaders》, 不详)：

类型 直觉 适合场景 平台关注点
map-style 可以通过 index 读取样本 图片分类、固定数据集、离线训练 shuffle、分片、缓存、随机种子
iterable-style 像流一样顺序产生样本 流式日志、超大语料、在线采样 去重、断点、worker 分片、可复现

这一区分很重要。分布式训练时，如果多个 worker 读取同一份 iterable-style 数据却没有正确切分，就可能重复训练同一批样本；如果 shuffle 和随机种子管理不好，不同实验之间也很难复现。

8.4 分布式训练时谁来切数据

单机训练里，DataLoader 慢通常是吞吐问题；分布式训练里，DataLoader 还会变成正确性问题。PyTorch 的 DistributedSampler 用于在分布式训练中把数据集切给多个进程，避免每个进程都遍历完整数据集 (《torch.utils.data.distributed.DistributedSampler》, 不详)。

平台工程要理解这个抽象：worker 数量改变时，数据分片规则也要跟着改变。

场景 如果分片错误 影响
每个 worker 都读全量数据 重复训练同一批样本 有效 batch 和 epoch 语义错误
有些样本永远读不到 数据覆盖不完整 收敛和评估失真
shuffle seed 不一致 实验无法复现 指标波动难解释
断点恢复不记录位置 resume 后重复或跳过样本 训练轨迹变化

这也是为什么训练平台要记录 world size、rank、sampler、seed、epoch 和数据版本。它们不是调试细节，而是训练语义的一部分。

8.5 DataLoader 慢时怎么判断

PyTorch performance tuning guide 提到，异步数据加载和多进程 DataLoader 可以减少主训练进程等待数据的时间，pin_memory=True 可以帮助更高效地把 CPU tensor 传到 GPU (《Performance Tuning Guide》, 不详)。

平台排障时，可以按下面顺序看：

  1. GPU 是否周期性空闲：如果 step 间有明显空档，先怀疑数据供给。
  2. CPU 是否打满：如果 CPU 忙，可能是解码、tokenizer、transform 太重。
  3. 存储吞吐是否稳定：远端对象存储、小文件、元数据请求都可能造成长尾。
  4. DataLoader worker 是否足够：太少喂不饱 GPU，太多会抢 CPU 和内存。
  5. batch 拼接是否昂贵：padding、变长序列和复杂 collate 会拖慢训练。

8.6 一个容易误判的案例：GPU 空等不是 GPU 问题

现象：一个文本训练任务 GPU 利用率只有 35%，但显存占用正常，batch size 也没有 OOM。

如果只看 GPU，会误以为模型太小或调度器分配不合理。按 DataLoader 链路拆开看，更可能是：

证据 可能原因 处理方向
CPU 一个核心打满 tokenizer 单线程瓶颈 并行 tokenizer、预分词、增加 worker
step 间隔周期性空档 DataLoader 预取不足 增加 num_workers、prefetch、缓存
存储延迟长尾明显 小文件或对象存储元数据慢 合并样本、使用 WebDataset/Parquet、节点本地缓存
H2D copy 等待明显 CPU 到 GPU 传输慢 pinned memory、异步拷贝、减少不必要拷贝

这类问题非常常见。平台要提供的不只是 GPU，还要提供“能喂饱 GPU 的数据路径”。

8.7 DataLoader 配置没有统一最优值

num_workers、prefetch、batch size、pin memory 这些参数都没有跨任务通用的最优值。它们取决于数据格式、CPU、存储、网络和模型计算速度。

参数 调大可能带来的好处 调大可能带来的代价
num_workers 并发读取和预处理更快 CPU 争抢、内存上涨、进程管理成本
prefetch 减少 GPU 等数据 内存占用增加，长尾样本可能堆积
batch size GPU 计算更成块 显存增加，padding 浪费可能变大
pinned memory CPU 到 GPU 传输更顺 host 内存压力增加

所以平台不应该只暴露静态默认值，而要提供可观测信号：DataLoader wait time、CPU 利用率、存储延迟、H2D copy 时间。没有这些指标，调参就是猜。

8.8 训练平台要提供什么

平台层面要支持：

  • 本地缓存，减少远端存储重复读取。
  • 并发 DataLoader，避免单进程喂不饱 GPU。
  • 数据分片，避免分布式 worker 读到重复样本。
  • 数据版本记录，保证实验可复现。
  • 数据加载指标，让训练慢的时候能定位是数据慢还是计算慢。
