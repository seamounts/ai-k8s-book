13  微调与继续训练

13.1 本章要回答的问题

不是每次训练都从随机初始化开始。很多生产任务是在已有模型基础上做 fine-tuning 或继续训练。

fine-tuning 的本质，是复用已有 parameters，在新数据、新目标或新领域上继续优化。

Excalidraw Style

fine-tuning：复用 base model，只改需要改的部分

手绘风格源文件：fig/excalidraw/finetune-map.excalidraw

Base base model 已有 weights，是所有微调产物的共同起点。
Data 领域/任务数据 决定模型要适配什么能力，也决定风险和许可证边界。
Train full 或 adapter full fine-tuning 更新全部 parameters；LoRA 等方法只训练少量附加参数。
Artifact 发布产物 可能是完整模型，也可能是 base model + adapter 的组合。
治理重点base model、adapter、数据版本和评测结果要能串起来。
资源重点训练全部 parameters 和只训练 adapter 的显存账本不同。
发布重点产物要能回滚、合并、审计，并与推理运行时兼容。

13.2 fine-tuning 和继续训练的区别

方式 本质 常见场景
fine-tuning 用新任务数据调整已有模型 领域适配、指令微调、分类头训练
continued pretraining 用新语料继续预训练 行业语料、语言扩展
LoRA / adapter 只训练少量附加参数 降低显存和训练成本
full fine-tuning 更新全部 parameters 资源充足、追求完整适配

Hugging Face PEFT 文档把 LoRA 放在参数高效微调方法里：使用 LoraConfig 包装模型后，可以只训练少量可训练参数 (《PEFT LoRA Developer Guide》, 不详)。对平台工程来说，LoRA 的重点不是“名字很新”，而是它改变了训练账本：base model weights 大多保持不变，adapter 参数和 optimizer state 变成主要训练状态。

13.3 系统关注点

微调不是“小训练”这么简单。平台仍然要关心：

  • base model 版本。
  • adapter 或 checkpoint 的关系。
  • 数据许可和数据版本。
  • 显存是否足够训练全部 parameters。
  • 产物如何合并、发布和回滚。
  • 多个 fine-tune 产物如何治理。

13.4 常见微调数据形态

不同微调目标，对数据格式和评估方式要求不同：

训练方式 数据形态 关注点
supervised fine-tuning prompt / response 回答格式、覆盖范围、拒答边界
instruction tuning 指令 / 输入 / 输出 指令多样性、任务分布、模板一致性
preference tuning chosen / rejected 偏好标注质量、冲突样本、评估集
continued pretraining 大规模无监督文本 数据清洗、重复率、领域分布、版权

平台要把这些数据形态和训练产物一起治理。一个 adapter 不能只记录文件路径，还要记录它来自哪份 base model、哪份数据、哪套 chat template、哪套评测。

13.5 full fine-tuning 和 LoRA 的系统差异

维度 full fine-tuning LoRA / adapter
更新范围 可能更新全部 parameters 主要更新 adapter 参数
显存压力 weights、activation、gradient、optimizer state 都重 optimizer state 明显更小，但仍有 activation
checkpoint 通常保存完整训练状态 需要保存 adapter、配置、base model 引用
发布形态 一个完整模型产物 base model + adapter，或合并后的模型
平台风险 成本高、恢复慢、产物大 版本关系复杂，adapter/base 不匹配会出错

所以微调平台要把 base model、adapter、data version、eval result 和 serving artifact 串成一条 lineage。否则“这个线上模型到底从哪个 base model 和哪批数据来”会变成难以回答的问题。

13.6 从训练产物到推理产物

微调完成不等于可以直接上线。至少要经过：

  1. 产物登记：记录 base model、adapter、tokenizer、config、训练代码版本。
  2. 离线评估：验证任务指标、通用能力、回归集和安全边界。
  3. 产物转换：必要时合并 LoRA、量化、转换格式或生成 serving artifact。
  4. 灰度发布：小流量验证延迟、错误率、输出质量和成本。
  5. 可回滚：保留上一版本 weights、adapter 和路由配置。

这里开始连接第三篇推理流程：训练产物进入推理服务后，会变成 model loading、weights 常驻、tokenizer 一致性和线上回滚问题。

13.7 本章小结

第二篇完成了训练流程闭环：数据进入系统，形成 batch，经过 forward/loss/backward 产生 gradient，optimizer 更新 parameters，指标记录训练进展，checkpoint 支撑恢复，fine-tuning 扩展已有模型。
