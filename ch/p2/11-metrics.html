11  训练指标与实验记录

11.1 本章要回答的问题

训练是否成功，不能只看进程有没有退出，也不能只看 GPU 是否很忙。训练系统至少要同时看算法指标、性能指标和系统指标。

Excalidraw Style

训练观测不是一条曲线，而是三块仪表盘

手绘风格源文件：fig/excalidraw/training-metrics.excalidraw

算法指标loss、accuracy、eval score：回答模型有没有变好。
性能指标step time、samples/sec、tokens/sec：回答 GPU 时间有没有变成训练进展。
系统指标显存、DataLoader 时间、NCCL 时间：回答瓶颈在哪里。

只看一类指标都会误判：loss 下降不代表资源高效；GPU 忙不代表模型变好；step 快不代表训练有效。

11.2 三类指标

指标类型 例子 回答什么问题
算法指标 loss、accuracy、eval score 模型是否在变好
性能指标 step time、samples/sec、tokens/sec GPU 时间是否转化为训练进展
系统指标 GPU 显存、GPU 利用率、DataLoader 时间、NCCL 时间 瓶颈在哪里

只看 loss，无法判断资源是否浪费；只看 GPU 利用率，无法判断模型是否收敛；只看 step time，无法判断训练是否有效。

PyTorch 的 TensorBoard 教程展示了如何把训练过程中的指标写入 TensorBoard，用曲线观察训练状态 (《Visualizing Models, Data, and Training with TensorBoard》, 不详)。但工具不是核心，核心是指标分层：算法、性能、系统三类信号要同时记录，才能把“模型有没有变好”和“集群有没有高效使用”分开判断。

11.3 实验记录要记录什么

一次训练至少要记录：

  • 代码版本。
  • 数据版本。
  • 模型配置。
  • hyperparameters。
  • checkpoint 路径。
  • loss 和 eval 指标曲线。
  • 运行环境和 GPU 型号。
  • 失败、恢复和重启事件。

这不是“做报表”，而是为了让结果可比较、可复现、可追责。

11.4 指标要能回答因果问题

好的训练记录不是“把所有指标都存下来”，而是能回答因果问题：

问题 至少需要哪些记录
为什么这次 loss 更低？ 代码版本、数据版本、模型配置、hyperparameters
为什么 step time 变慢？ batch shape、DataLoader 时间、GPU kernel 时间、NCCL 时间
为什么恢复后曲线变了？ checkpoint、optimizer state、scheduler state、random state
为什么线上效果不如 eval？ eval 数据版本、线上数据分布、推理参数、模型版本

如果实验记录不能回答这些问题，它就只是日志，不是工程资产。

11.5 一个判断例子

现象 不能直接下结论 更合理的判断
loss 下降很快 训练一定成功 可能数据泄漏，要看 eval 和数据版本
GPU 利用率 95% 资源一定高效 可能算子低效或通信等待，要看 tokens/sec 和 profiler
step time 变短 模型训练更好 可能 batch 变小了，要看 samples/sec 和收敛
eval score 变好 可以上线 还要看线上分布、延迟、成本和安全评测

11.6 平台应该提供的训练看板

训练平台至少应该把三类信号放在同一个视图里：

看板区域 关键指标 目的
训练进展 loss、eval score、learning rate、step 判断模型是否有效推进
资源效率 samples/sec、tokens/sec、step time、GPU 利用率 判断 GPU 时间有没有转化为训练进展
系统瓶颈 DataLoader wait、H2D copy、NCCL time、checkpoint time、OOM 判断瓶颈在哪里

这三块必须能按 step 对齐。否则看到 GPU 利用率下降，却不知道当时是不是在 checkpoint；看到 loss 抖动，也不知道是不是数据版本或学习率发生变化。

11.7 本章小结

训练指标要同时回答三个问题：模型有没有变好，资源有没有有效使用，系统瓶颈在哪里。
