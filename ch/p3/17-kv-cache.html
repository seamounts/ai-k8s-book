17  KV Cache 与显存占用

17.1 本章要回答的问题

LLM 推理里，KV Cache 往往决定并发上限。它不是可有可无的缓存，而是逐 token 生成的核心状态。

17.2 KV Cache 是什么

Transformer attention 会为 token 计算 key 和 value。生成新 token 时，模型需要参考历史 token。如果每一步都重新计算全部历史 token，成本会非常高。KV Cache 保存历史 token 的 key/value，让 decode 可以复用已有结果。Hugging Face Transformers 文档也把 KV cache 作为 generation 加速的核心机制，并说明 cache 会避免重复计算已处理 token 的 key/value (《Transformers KV cache strategies》, 不详)。

可以把它理解成：

历史上下文 -> key/value 中间状态 -> 后续 token 生成反复读取

17.3 为什么它吃显存

KV Cache 会随这些因素增长：

  • batch 中请求数量。
  • 每个请求的上下文长度。
  • 模型层数。
  • hidden size / attention head 数。
  • dtype。

这解释了一个重要现象：同一个模型，短 prompt、低并发时很稳；长上下文、高并发时突然 OOM。

粗略理解可以用这个乘法关系：

KV Cache 规模 ∝ 并发请求数 x 上下文长度 x 模型层数 x hidden size x dtype 字节数

真实实现还会受 attention head、分组查询、block size、分页管理和运行时 buffer 影响，但平台判断先抓住这个乘法关系就够了：并发和上下文长度一起上升时，显存会很快被吃掉。

17.4 平台要关注什么

问题 影响
最大上下文长度 决定单请求最坏显存
并发请求数 决定总 KV Cache
长短请求混合 造成显存碎片和调度复杂度
cache 回收 请求结束后能否及时释放

vLLM 等 LLM 推理运行时的重要价值之一，就是围绕 KV Cache 做更细粒度的显存管理和请求调度 (《vLLM Documentation》, 不详)。NVIDIA TensorRT-LLM 也把 KV cache reuse 作为高级优化主题，用来减少重复 prefill 计算并提升共享前缀场景下的效率 (《TensorRT-LLM KV Cache Reuse》, 不详)。

vLLM 的 PagedAttention 文档把 attention 的 KV cache 管理作为核心设计主题 (《vLLM Paged Attention》, 不详)。从平台角度看，PagedAttention 的直觉很像操作系统分页：不要把每个请求的 KV Cache 当作一整块连续大内存，而是用更细粒度的块来管理，从而提升显存利用率并减少碎片影响。

17.5 为什么 prefix 复用有价值

很多生产请求有相同系统提示词、相同工具说明或相同文档前缀。如果每个请求都重新 prefill 这段公共前缀，会浪费 GPU 时间。prefix caching 或 KV cache reuse 的直觉是：相同前缀已经计算过的 key/value，可以被后续请求复用。

vLLM 也提供 Automatic Prefix Caching，用于在请求之间复用已计算的前缀 KV Cache (《vLLM Automatic Prefix Caching》, 不详)。它的价值不是“多一个缓存开关”，而是把 prompt 结构、请求路由和显存管理联系起来：相同前缀越多，复用收益越明显；但 cache 越多，也越需要淘汰和隔离策略。

但它也带来新的平台问题：

  • 如何判断哪些前缀可以复用？
  • cache 占用显存，何时淘汰？
  • 多租户之间是否允许共享 cache？
  • 模型版本、tokenizer 和参数不一致时是否会误复用？

所以 KV Cache 优化不是单纯 runtime 技巧，也涉及隔离、版本和治理。

17.6 KV Cache 的生命周期

一个请求的 KV Cache 会经历四个阶段：

阶段 发生什么 平台关注点
分配 请求进入 prefill 前申请 cache 空间 是否超过上下文和并发配额
填充 prefill 写入 prompt 的 key/value 长 prompt 会快速消耗显存
追加 decode 每生成一个 token 就追加新 key/value 输出越长，占用越久
回收 请求完成、取消或超时后释放 是否及时释放，是否产生碎片

这也是为什么请求取消很重要。用户断开连接后，如果 runtime 没有及时取消 decode 并释放 KV Cache，GPU 仍然可能继续为一个没人接收的请求工作。

17.7 一个 OOM 案例

一个服务在 4k 上下文、并发 16 时稳定；上线 32k 上下文后，并发还没到 16 就 OOM。原因往往不是 weights 变大了，而是 KV Cache 被放大了。

排查时要看：

信号 说明
OOM 出现在长 prompt 或长输出请求 KV Cache 很可能是主因
显存随活跃请求数线性上升 并发请求在占用 cache
请求结束后显存没有及时下降 cache 回收、碎片或 runtime 池化策略需要检查
P99 延迟和 OOM 同时变差 队列和显存容量已经互相影响

17.8 KV Cache 需要配额

多租户推理平台不能只按请求数限流，还要按上下文和输出长度限流。更合理的准入控制会同时看：

  • prompt tokens。
  • max output tokens。
  • 当前活跃请求数。
  • 当前 KV Cache 使用量。
  • 租户优先级和配额。

否则一个租户少量超长上下文请求，就可能挤掉其他租户大量短请求。

17.9 本章小结

KV Cache 是 LLM 推理区别于普通模型推理的核心状态。它把推理服务从“无状态请求处理”变成“显存状态管理系统”。
