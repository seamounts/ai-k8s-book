19  延迟、吞吐与并发

19.1 本章要回答的问题

推理性能不能只看 requests/sec，也不能只看 GPU 利用率。在线服务的性能是延迟、吞吐、并发和成本的组合。

Excalidraw Style

推理指标时间线：TTFT、TPOT、P95/P99 看哪里

手绘风格源文件：fig/excalidraw/latency-metrics.excalidraw

Queue 排队等待 负载高、batching 等待、限流都会拉长这一段。
Prefill 首 token 前计算 prompt 长度、tokenizer、prefill batch 影响 TTFT。
First Token TTFT 截止点 用户第一次看到系统回应。
Decode 后续 token 每个 token 的间隔就是 TPOT/ITL 关注的区域。
TTFT交互体感第一指标，尤其影响聊天和代码补全。
TPOT输出速度指标，影响长文本生成体验。
P95/P99看长尾，不要只看平均值。

19.2 核心指标

指标 含义 适用场景
TTFT 从请求进入到第一个 token 返回 聊天、代码补全、流式输出
TPOT 每个输出 token 的平均时间 LLM decode 速度
P95/P99 长尾延迟 SLA 和用户体验
tokens/sec 单副本或集群生成吞吐 容量规划
concurrency 同时处理请求数 服务承载能力
每 token 成本 成本效率 GPU 成本治理

这里补一个术语统一：本书使用 TTFT 表示 Time To First Token，使用 TPOT 表示 Time Per Output Token。有些系统也会用 ITL（Inter-Token Latency）描述相邻输出 token 的间隔。三者不是互斥指标，而是从不同角度观察同一条生成时间线。

19.3 指标之间会冲突

提高 batch size 可能提高 tokens/sec，但也可能增加 TTFT。提高并发可以提高 GPU 利用率，但 KV Cache 会挤占显存。降低延迟可能需要更多副本，但成本会上升。

因此，推理优化必须先确定目标：

  • 交互式服务优先 TTFT 和 P95。
  • 离线批推理优先吞吐和成本。
  • 长文本生成要关注 TPOT 和 KV Cache。
  • 多租户服务要关注隔离和公平性。

19.4 四类服务的目标不同

服务类型 优先目标 典型策略
聊天助手 TTFT、P95、稳定 streaming 控制 queue delay，保留低延迟副本
代码补全 极低 TTFT、短输出 小 batch、快速取消、靠近用户
离线批推理 tokens/sec、成本 大 batch、充分排队、低优先级资源
长文生成 TPOT、上下文容量 KV Cache 管理、长短请求隔离

这就是为什么“提高 GPU 利用率”不是一个完整目标。GPU 利用率只是信号，真正的优化目标必须和服务形态绑定。

19.5 一个容量估算例子

假设一个副本平均可以输出 2,000 tokens/sec。业务请求平均输出 200 tokens，目标是每秒处理 5 个请求。

粗略吞吐需求是：

5 requests/sec x 200 tokens/request = 1,000 tokens/sec

看起来一个副本够。但这只是平均吞吐，还没有考虑：

  • prompt prefill 对 TTFT 的影响。
  • P95/P99 的长尾。
  • 高峰流量。
  • KV Cache 显存上限。
  • 副本故障和滚动升级冗余。

生产容量规划不能只用平均 tokens/sec，要同时用延迟目标、并发、上下文长度和可用性目标校验。

19.6 SLO 要分层定义

推理服务的 SLO 不应该只写“平均延迟低于多少”。更合理的是分层：

层级 示例 SLO 为什么
请求准入 超过上下文上限的请求在 100ms 内拒绝 快速失败比排队后失败更好
首 token TTFT P95 < 1.5s 交互体感核心
生成速度 TPOT P95 < 80ms 长文本输出体验
可用性 5xx rate < 0.1% 服务稳定性
成本 每 1k tokens 成本低于预算 GPU 成本治理

SLO 一旦定义清楚，扩缩容、batching、限流和降级才有判断标准。否则“优化推理性能”会变成没有方向的调参。

19.7 load test 要覆盖长尾

压测不能只用平均 prompt 和平均输出长度。至少要覆盖：

  • 短 prompt + 短输出。
  • 短 prompt + 长输出。
  • 长 prompt + 短输出。
  • 长 prompt + 长输出。
  • 多租户混合。
  • streaming 连接长时间保持。

因为 LLM 推理的瓶颈经常出现在长尾：少量长上下文请求就能改变 KV Cache 占用和 P99。

19.8 本章小结

推理服务不是单指标优化。后面讲推理运行时、模型服务框架和扩缩容时，都要先说明优化目标是什么。
