16  分词、上下文处理与逐 token 生成

16.1 本章要回答的问题

LLM 推理不是一次性输出完整文本，而是通常按 token 逐步生成。理解这个过程，才能理解 TTFT、TPOT、KV Cache 和 streaming。

16.2 从文本到 token

用户输入的是文本，模型处理的是 token id。tokenizer 会把文本拆成 token，再映射成整数序列。这个过程通常在 CPU 上发生，也可能成为高并发服务的瓶颈。

text -> tokenizer -> token ids -> embedding -> model

Hugging Face Transformers 文档把 tokenizer 作为文本进入模型前的重要组件：文本需要先被 tokenizer 转换成模型可处理的输入 (《Transformers Tokenizer Summary》, 不详)。这对平台有直接含义：tokenizer 版本必须和 weights/config 匹配，否则同一段文本会被切成不同 token，模型行为就会变。

Excalidraw Style

prefill 与 decode：一次 LLM 请求的两种计算形态

Prefill：先读完整 prompt

prompt长上下文一次性进入
compute大块矩阵计算

Decode：每次生成下一个 token

read cache读取历史 key/value
next token小步循环计算
append把新 token 写回 cache

TTFT 主要被排队、tokenizer 和 prefill 影响；TPOT 主要被 decode 调度、KV Cache 访问和 batch 形态影响。

16.3 prefill 和 decode

LLM 推理常分成两个阶段：

阶段 做什么 性能特点
prefill 处理完整 prompt，建立初始 KV Cache 计算量大，适合较大 batch
decode 每次生成一个或少量 token 强依赖 KV Cache，延迟敏感

TTFT 主要受排队、tokenizer、prefill 和首次调度影响。TPOT 主要受 decode 循环、KV Cache 访问和 batch 调度影响。

可以把一次请求的耗时拆成：

TTFT = queue time + tokenize time + prefill time + first scheduling time
TPOT = decode loop time / output tokens

这不是精确公式，而是排障公式。TTFT 高，要优先看队列、tokenizer、prefill；TPOT 高，要优先看 decode batch、KV Cache、显存带宽和运行时调度。

16.4 streaming 为什么重要

流式返回不是为了炫技，而是改善用户体感。即使完整生成需要十几秒，只要首 token 很快出现，用户会觉得系统在响应。

但 streaming 也带来系统压力：

  • 请求持续占用连接。
  • decode 需要多轮调度。
  • 不同请求生成长度不同。
  • batch 中请求会不断进出。

16.5 一个例子：短 prompt 和长 prompt 为什么体验不同

两个用户同时请求同一个模型：

请求 prompt 长度 max tokens 主要压力
A 50 tokens 200 tokens prefill 轻，decode 时间占比高
B 8,000 tokens 200 tokens prefill 重，首 token 可能很慢

如果系统把 A 和 B 粗暴放在同一个调度节奏里，A 可能被 B 的长 prefill 拖慢。生产 runtime 会尝试通过调度策略、batching 策略甚至 prefill/decode 分离来降低这种相互影响。后面讲推理运行时和多副本架构时，会继续展开这件事。

16.6 sampling 参数也属于服务语义

LLM 生成不只是“算出下一个 token”。runtime 还要根据 generation config 决定如何采样。Hugging Face 文本生成文档中会讨论 greedy、sampling、beam search 等生成策略 (《Transformers Text Generation Strategies》, 不详)。生产服务里常见参数包括：

参数 作用 平台影响
max_tokens 最多生成多少 token 决定请求最长占用时间和 KV Cache 增长
temperature 控制随机性 影响结果稳定性和可复现
top_p / top_k 限制候选 token 范围 影响生成质量和计算路径
stop 遇到指定字符串停止 影响 streaming 结束条件
seed 控制随机采样复现 调试和评估时很重要

这些参数不是业务层随便透传就完了。平台要限制最大值、记录请求参数、在灰度和回滚时保持默认值一致，否则性能和结果都会变得不可解释。

16.7 本章小结

LLM 推理的关键路径是 tokenizer、prefill、decode 和 streaming。下一章会专门讲 decode 为什么离不开 KV Cache。
