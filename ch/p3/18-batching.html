18  batching、队列与流式返回

18.1 本章要回答的问题

推理服务为什么要 batching？因为单个请求往往无法充分利用 GPU。多个请求合成一个 batch，可以提高吞吐。但 batching 会引入等待时间，所以它必须和延迟目标一起设计。

Triton Inference Server 的 dynamic batching 文档说明，动态批处理会把独立推理请求合并成批，从而提升吞吐；但它也需要配置最大 batch size、队列延迟等参数 (《Triton Inference Server: Dynamic Batcher》, 不详)。

Excalidraw Style

continuous batching：decode 队伍边跑边换人

手绘风格源文件：fig/excalidraw/continuous-batching.excalidraw

t0 A / B / C 入队 多个请求一起 prefill 或进入 decode。
t1 A 生成中 B 结束释放 cache，C 仍在生成。
t2 D 插队加入 新请求不必等整个 batch 完成，可以加入后续 decode。
t3 GPU 保持忙碌 吞吐提高，但调度和显存管理更复杂。

batching 的本质是用等待换吞吐；continuous batching 进一步把“固定一批一起跑完”变成“token 级动态编队”。

18.2 三种 batching

类型 做法 适合场景
固定 batching 凑够固定数量再执行 离线批推理
动态 batching 短时间窗口内合并请求 在线服务吞吐优化
continuous batching decode 过程中持续加入/移除请求 LLM 推理

LLM 的 batching 更复杂，因为每个请求的 prompt 长度、生成长度和当前 decode 位置都不同。一个 batch 不是一次性执行完，而是在 token 级别不断变化。

可以这样区分：

模式 像什么 优点 风险
固定 batching 等一车人坐满再发车 简单、吞吐可控 在线请求等待时间不可控
dynamic batching 在很短窗口里拼车 兼顾吞吐和延迟 queue delay 配不好会伤 TTFT
continuous batching 车不停车，乘客边下边上 LLM decode 吞吐更好 runtime 调度和 KV Cache 管理复杂

18.3 dynamic batching 的关键参数

Triton dynamic batching 给平台工程一个很好的抽象：吞吐不是免费来的，需要用排队时间换 batch 规模 (《Triton Inference Server: Dynamic Batcher》, 不详)。

参数 含义 影响
max batch size 单次执行最多合并多少请求 越大越可能提高吞吐，也越可能增加显存
queue delay 最多等待多久来凑 batch 越长越容易凑批，也越影响延迟
preferred batch size 倾向形成的 batch 大小 用于匹配硬件或模型最优形状

在线服务最容易犯的错，是只调大 batch size，却没有给 TTFT/P95/P99 设置硬约束。

18.4 一个调参例子

假设当前服务 tokens/sec 不高，GPU 利用率也不高。你把 max batch size 从 8 调到 32 后，tokens/sec 上去了，但用户投诉首 token 变慢。

这说明 batching 起作用了，但队列代价也变大了。下一步不是继续调大 batch，而是同时看：

指标 如果变差，说明什么
TTFT 请求为了凑 batch 等太久，或 prefill 被大 batch 拖慢
TPOT decode batch 太大，单轮 token 间隔变长
P99 长请求、短请求互相干扰，或队列长尾扩大
OOM / eviction batch 和 KV Cache 挤占显存

推理 batching 的底线是 SLA。吞吐提升如果换来不可接受的长尾延迟，就是错误优化。

18.5 队列不是越短越好

没有队列，GPU 可能吃不满；队列太长，用户延迟变差。推理系统要控制：

  • 最大排队时间。
  • 最大 batch size。
  • 优先级。
  • 超时和取消。
  • streaming 的持续占用。

18.6 队列要有准入和取消

在线推理队列不是普通 FIFO。它至少要支持：

能力 为什么需要
准入控制 请求进入前判断上下文长度、max tokens、租户配额
优先级 交互式请求和离线批请求不能完全同权
超时 过期请求继续排队只会浪费资源
取消 客户端断开后要释放队列项和 KV Cache
公平性 防止某个租户或长请求长期占满 batch

很多推理事故不是 runtime 算错了，而是队列语义太弱。队列没有准入，就会把不可服务的请求放进系统；队列没有取消，就会为已经无意义的请求继续消耗 GPU。

18.7 长短请求混部的问题

短请求和长请求混在一个队列里，会出现两个问题：

问题 表现
head-of-line blocking 长 prefill 请求挡住短请求，短请求 TTFT 变差
cache pressure 长上下文请求占住大量 KV Cache，短请求并发下降

生产系统通常会用模型副本池、优先级队列、上下文长度分桶或 prefill/decode 分离来缓解。后面进入推理架构时，这些策略会变成具体部署方案。

18.8 本章小结

batching 是吞吐和延迟之间的调节器。训练里 batch 主要服务吞吐和收敛；推理里 batching 必须同时服务吞吐、TTFT、TPOT 和用户体感。

《Triton Inference Server: Dynamic Batcher》. 不详. https://docs.nvidia.com/deeplearning/triton-inference-server/user-guide/docs/user_guide/batcher.html.
