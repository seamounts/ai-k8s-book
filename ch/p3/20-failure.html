20  推理服务的失败模式

20.1 本章要回答的问题

推理服务失败不只是进程 crash。在线系统的失败可能表现为超时、长尾延迟、OOM、模型加载失败、结果错误、流式中断或版本混乱。

Excalidraw Style

推理故障地图：先定位是哪一段坏了

手绘风格源文件：fig/excalidraw/inference-failures.excalidraw

加载失败artifact、tokenizer、config、adapter 或权限问题。
容量失败KV Cache、batch、长上下文导致 OOM 或驱逐。
延迟失败队列、prefill 拥塞、decode 过慢、P99 变差。
流式失败连接断开、网关超时、runtime 异常。
版本失败weights、tokenizer、template、adapter 不匹配。
治理失败限流、回滚、灰度、观测信号缺失。

线上排障不要只问“Pod 活着吗”。要问请求卡在加载、队列、prefill、decode、streaming、版本还是治理层。

20.2 常见失败模式

失败模式 可能原因 影响
冷启动慢 weights 下载和加载慢 扩容不及时
OOM KV Cache、batch 过大、长上下文 请求失败或副本重启
TTFT 抖动 队列过长、prefill 拥塞 用户体感差
TPOT 变慢 decode batch 过大、显存带宽压力 输出变慢
流式中断 连接、网关、runtime 异常 用户请求失败
版本错配 tokenizer、weights、配置不一致 结果异常

20.3 线上排障顺序

推理服务排障不要先看代码，也不要只看 Pod 是否 Running。建议按请求路径排：

  1. admission：请求是否被限流、拒绝、路由到错误模型？
  2. loading：副本是否真的 ready？weights、tokenizer、adapter 是否加载一致？
  3. queue：排队时间是否变长？是否有租户或长请求占满队列？
  4. prefill：长 prompt 是否把首 token 延迟打高？
  5. decode：TPOT 是否变慢？KV Cache 是否接近显存上限？
  6. streaming：网关、客户端、runtime 哪一段断开？
  7. control：灰度、回滚、扩缩容、熔断是否按预期生效？

这个顺序能把“服务慢”拆成可验证的假设，而不是在 GPU、网关、runtime 和业务代码之间来回猜。

20.4 生产服务要有的控制面能力

推理平台至少需要：

  • 健康检查。
  • 模型预热。
  • 灰度和回滚。
  • 超时与取消。
  • 限流和熔断。
  • 请求级日志和指标。
  • 模型版本、tokenizer 版本和配置一致性校验。

20.5 三个典型事故

事故 根因 应该补的能力
扩容后仍然超时 新副本模型还没加载完成就接流量 ready gate、warmup、模型缓存
白天稳定，晚上 OOM 夜间批量任务使用长上下文和大 max tokens 请求准入、上下文配额、KV Cache 观测
灰度结果异常 新 weights 配了旧 tokenizer 或旧 chat template artifact 一致性校验、版本绑定、回滚

这些事故的共同点是：进程可能都还活着，但服务语义已经坏了。推理平台必须观测请求级、模型级和资源级信号。

20.6 推理服务的降级策略

不是所有故障都只能扩容。生产系统要准备降级手段：

压力来源 可用降级
队列过长 拒绝低优先级请求、缩短 queue delay、返回 busy
KV Cache 紧张 降低 max context、降低 max output tokens、限制长请求
TTFT 过高 分离长 prompt 队列，减少 batching 等待
TPOT 过高 降低并发，切换更小模型或量化模型
冷启动过慢 使用预热池，延迟接流量，保留最小副本数

降级不是失败，而是把系统从不可控状态拉回可控状态。没有降级策略的推理平台，往往只能在超时、OOM 和重启之间被动摇摆。

20.7 故障复盘要追三条线

推理故障复盘至少追三条线：

  1. 请求线：哪些请求、哪个租户、什么上下文长度、什么生成参数。
  2. 模型线：哪个 weights、tokenizer、adapter、template、runtime 版本。
  3. 资源线：当时的 GPU、显存、KV Cache、队列、batch、网络和扩缩容事件。

三条线合在一起，才能回答“为什么这一次服务坏了”。只看 Pod 日志，很容易错过模型版本、请求形态和显存状态之间的关系。
